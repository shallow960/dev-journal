# 2026-07-16 코드를 합치면 서버에 자동으로 배포되게 만들기

> 태그: #새기술 #배포 #CICD

## 무엇을 / 왜
한동안 만든 코드가 저장소에는 쌓였는데, 정작 실제 서버에는 반영이 안 돼 있었다.
반영하려면 매번 손으로 앱 이미지를 만들어 서버에 올려야 했는데 번거롭고 실수도 잦았다.
"코드를 합치면 알아서 서버에 올라가면 좋겠다" 싶어 자동 배포(CI/CD)를 처음 만들어봤다.

## 어떻게 풀었나
저장소에 코드가 들어오면 자동화 도구(러너)가 이 순서로 돌게 엮었다.
- 앱 이미지를 만든다(빌드).
- 만든 이미지를 서버로 보낸다.
- 서버에서 옛 버전을 새 이미지로 교체하고 앱을 다시 띄운다.

몇 가지 걸림돌이 있었다.
- 서버가 저사양이라 무거운 빌드를 못 버텼다. 그래서 빌드는 힘 좋은 자동화 서버에서 하고, 완성된
  결과물만 실제 서버로 보내도록 역할을 나눴다.
- 자동화 도구가 컨테이너 안에서 도는 방식이라, 그 안에서 이미지를 만들려면 "컨테이너 안의 컨테이너"를
  켜야 했다. 태그 이름만 보고 실행 방식을 짐작했다가 틀렸고, 아주 작은 확인 작업을 먼저 돌려 실제
  환경을 눈으로 검증하고 나서야 정확히 알 수 있었다.
- 배포가 마지막 "서버 접속" 단계에서 계속 실패했다. 접속 키가 "형식 오류"라고 떠서 처음엔 키 파일이
  깨진 줄 알았는데, 원인이 두 개 겹쳐 있었다. ① 컴퓨터에서 복사한 키가 줄바꿈 문자 차이로 손상된 것
  (키를 그대로 붙여넣지 않고 안전한 문자열로 바꿔 전달해 해결), ② 넣어둔 비밀값이 "보호된 곳에서만
  보이는" 설정 탓에 시험용 환경에서는 통째로 빈 값이던 것(그 설정을 꺼서 해결).
- 매번 몇 분씩 걸리는 전체 빌드를 기다리는 대신, 문제 구간만 수십 초에 확인하는 최소 진단 작업을 따로
  만들어 원인을 빠르게 좁혔다.

## 💡 배운 것
- "합쳤다"와 "서버에 반영됐다"는 완전히 다른 단계다. 저장소에 코드가 들어가는 것과, 그 코드가 실제
  서버에서 도는 것 사이를 이어주는 게 배포이고, 그 다리를 자동화한 게 CI/CD였다.
- 실행 환경은 짐작하지 말고 작은 확인 작업으로 실제로 돌려봐야 한다. 이름표(태그)만 보고 넘겨짚으면
  틀린다.
- "형식 오류" 같은 표면 메시지에 갇히면 엉뚱한 데만 판다. "키가 깨진 것"과 "키가 아예 전달 안 된 것"은
  증상이 같아 보여도 원인이 다르다. 확인 가능한 작은 실험으로 하나씩 갈라내는 게 훨씬 빨랐다.

## 코드로 보기 (구조 골격)
"합치면 = 빌드 → 전송 → 재기동" 흐름을 값은 더미로, 구조만 재현한다.

```yaml
# 목적: 특정 브랜치에 코드가 올라오면 자동으로 빌드해 서버에 반영한다.
stages: [deploy]

deploy:
  image: docker:27
  services: [docker:27-dind]              # 컨테이너 안에서 이미지를 만들기 위한 "안의 컨테이너"
  rules:
    - if: '$CI_COMMIT_BRANCH == "main"'    # main 에 올라오면 자동 실행 (그 외엔 수동)
  script:
    # 1) 힘 좋은 러너에서 이미지 빌드 (저사양 서버는 빌드를 안 시킨다)
    - docker build -t app-backend:latest ./backend
    - docker build -t app-frontend:latest ./frontend
    # 2) 이미지를 한 덩어리로 압축 저장
    - docker save app-backend:latest app-frontend:latest | gzip > images.tgz
    # 3) 접속 키 복원 - 붙여넣기로 깨지지 않게 base64 로 저장해 둔 값을 되돌린다
    - printf '%s' "$SSH_KEY_B64" | base64 -d > key && chmod 600 key
    # 4) 서버로 전송하고, 서버에서 교체 + 재기동
    - scp -i key images.tgz "$DEPLOY_USER@$DEPLOY_HOST:~/app/"
    - ssh -i key "$DEPLOY_USER@$DEPLOY_HOST"
        'cd ~/app && gunzip -f images.tgz && docker load -i images.tar && docker compose up -d'
```

## 소감
막상 되고 나니 별것 아닌 것 같지만, 만드는 내내 "왜 안 되지"의 연속이었다. 특히 접속 키 문제는 표면
메시지 하나에 붙잡혀 한참 헤맸는데, 매번 전체를 다시 돌리지 않고 문제 지점만 콕 찍어 확인하는 작은
장치를 만든 뒤로는 원인 찾는 속도가 확 빨라졌다. 앞으로는 코드만 합치면 서버에 알아서 올라간다는 게,
번거로운 반복 하나를 확실히 줄인 것 같아 뿌듯했다.
