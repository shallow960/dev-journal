# 2026-07-02 세션 없이, 외부 결제창 콜백에서 사용자를 안전하게 찾기

> 태그: #새기술 #보안 #결제

## 무엇을 / 왜
카드 등록 결제창은 내 로그인 세션 밖에서 열리고, 처리가 끝나면 결과를 내 서버로 되돌려준다. 이때 "이 결과가 누구 것인지"를 알아야 하는데, 그 시점엔 로그인 세션이 붙어 있지 않았다.

## 어떻게 풀었나
결제창을 띄울 때, 사용자 식별자에 서버만 아는 비밀키로 만든 서명(HMAC)을 붙여 함께 넘겼다.

결과가 돌아오면, 되돌아온 식별자와 서명을 서버가 다시 계산해 맞는지 검증했다. 서명이 일치해야만 그 사용자로 인정한다.

별도의 임시 저장소(세션 테이블·캐시)를 두지 않고, 값을 왕복시키는 것만으로 신원을 이어붙였다. 서명이 있어서 값을 위조하거나 남의 것으로 바꿔치기할 수 없다.

## 💡 배운 것
- 세션이 끊기는 외부 리다이렉트 흐름에선, 상태를 서버에 저장하는 대신 "서명된 값을 왕복"시키는 방식이 단순하고 안전하다.
- 서명(HMAC)은 값을 "숨기는" 게 아니라 값이 "위조되지 않았음을 보장"하는 도구다. 식별자는 그대로 보이되, 바꾸면 서명 검증에서 걸린다.

## 코드로 보기 (개념 재현 테스트)
실제 서비스 코드는 키·내부 구조가 얽혀 공개하기 어려워, 핵심 개념만 자기완결로 재현한 테스트로 대신한다.

```java
// 목적: "세션 없이 넘긴 사용자 식별자"가 위조되지 않았는지 서명으로 검증하는 로직 테스트.
class StatelessCallbackTokenTest {

    // 서버만 아는 비밀키 (실제로는 시크릿/환경변수에서 주입, 여기선 테스트용 더미)
    private static final String SECRET = "test-secret-key";

    // 사용자 식별자에 HMAC 서명을 붙여 "왕복용 토큰"을 만든다 -> "<id>.<서명>"
    private String issue(String userId) {
        String sig = hmacSha256Hex(userId, SECRET);
        return userId + "." + sig;              // 이 값을 결제창에 실어 보냄
    }

    // 돌아온 토큰의 서명을 다시 계산해 일치할 때만 사용자 id를 돌려준다. 아니면 예외.
    private String verify(String token) {
        int dot    = token.lastIndexOf('.');
        String id  = token.substring(0, dot);   // 앞부분 = 사용자 식별자
        String sig = token.substring(dot + 1);  // 뒷부분 = 서명
        // 같은 키로 다시 계산한 서명과 비교 -> 위조 여부 판별 (타이밍 공격 방지 비교)
        if (!constantTimeEquals(hmacSha256Hex(id, SECRET), sig)) {
            throw new IllegalArgumentException("서명 불일치: 위조된 토큰");
        }
        return id;
    }

    @Test
    void 정상_토큰은_원래_사용자를_복원한다() {
        String token = issue("user-123");                 // 보낼 때
        assertThat(verify(token)).isEqualTo("user-123");  // 콜백에서 그대로 복원
    }

    @Test
    void 식별자를_바꿔치기하면_거부한다() {
        String token = issue("user-123");
        // 공격자가 뒤 서명은 그대로 두고 앞 사용자 id만 남의 것으로 교체
        String tampered = "user-999." + token.substring(token.indexOf('.') + 1);
        // 서명이 안 맞으므로 검증에서 반드시 막혀야 한다
        assertThatThrownBy(() -> verify(tampered))
            .isInstanceOf(IllegalArgumentException.class);
    }
}
```

## 소감
처음엔 "임시로 어디 저장해둘까"를 고민했는데, 서명 왕복으로 저장 없이 깔끔하게 풀렸다. 상태를 만들지 않는 설계가 관리 포인트를 줄인다는 걸 체감했다.
