# 2026-07-01 NICEPay 연동

> 태그: #새기술 #트러블슈팅 #결제

## 오늘 한 일 (요약)
- 쇼핑몰에 **실제 카드 결제(NICEPay)** 연동을 시작 - 결제창 띄우기부터 서버 최종 승인까지 흐름을 이었다.
- (별개로) 쇼핑몰 화면 구조 리팩터링을 마무리했다.

## 무엇을 / 왜
"결제"를 직접 붙여본 건 처음이다. 손님이 카드로 내는 돈은 우리가 직접 받는 게 아니라
결제대행사(PG)를 거친다. 그래서 결제는 늘 **① 결제창 요청 → ② 손님 인증 → ③ 결제사 통보 →
④ 우리 서버가 진짜인지·금액 맞는지 확인하고 승인** 이라는 4단계로 흐른다는 걸 몸으로 이해했다.

## 어떻게 풀었나 - 기억에 남는 문제 하나
**"기존 결제 보안 검사가 새 결제사를 통째로 막아버릴 뻔한 문제"**

우리 시스템엔 결제 위조·중복 요청을 막는 공통 보안 검사가 있는데, 기존 결제사들이 주던
특정 식별값을 *반드시* 요구하도록 만들어져 있었다. 그런데 새로 붙이는 결제사는 그 값을
주지 않아서, 그대로 두면 정상 결제까지 전부 차단될 상황이었다.

공통 보안 검사를 뜯어고치면 다른 결제사들이 위험해진다. 그래서 **결제 결과를 받는 지점에서,
결제사가 준 "거래번호"를 그 식별값 자리에 대신 채워** 통과시켰다.
- 중복 차단 → 거래번호가 1회성이라 그대로 보장됨
- 위조 방지 → 어차피 별도의 서명 검증이 담당함

결과적으로 **기존 코드도, 다른 결제사도 건드리지 않고** 보안도 안 약해지게 풀렸다.

## 💡 배운 것
- **외부 연동은 "공통 모듈을 고치기"보다 "경계에서 차이를 흡수하기"를 먼저 떠올리자.**
  결제사마다 규격이 달라도, 연결 지점 한 곳에서 맞춰주면 안쪽 코드는 그대로 둘 수 있다.
- **문서를 믿기 전에 값으로 증명하자.** 결제사 보안 서명이 문서 설명대로 계산되는지,
  문서에 나온 예시 값으로 직접 계산해 결과가 똑같이 나오는 걸 확인한 뒤에야 구현을 신뢰했다.
  "읽었다"와 "검증했다"는 다르다.
- **"콜백=서버 웹훅"이라고 넘겨짚지 말자.** 이 결제사의 결과 전송은 사실 손님 브라우저가
  보내는 것이라, 외부 터널(ngrok) 없이 로컬에서도 받을 수 있었다. 흐름을 정확히 그려야
  괜한 삽질을 안 한다.

## 코드로 보기 (개념 재현 테스트)
"문서를 읽었다"가 아니라 "예시 값으로 계산해 결과가 맞다"를 코드로 못박은 그 지점을 재현한다.

```java
// 목적: 결제사 보안 서명(SignData) 공식이 규격대로 계산되는지 더미 값으로 검증.
class PaymentSignatureTest {

    // 서명 규칙(예시): SHA-256( 여러 필드를 정해진 순서로 이어붙임 + 비밀키 )
    // 필드 순서/구성이 규격과 한 글자라도 다르면 결제사가 거부한다.
    private String sign(String ediDate, String mid, String amount, String key) {
        String plain = ediDate + mid + amount + key;   // 규격이 정한 결합 순서
        return sha256Hex(plain);
    }

    @Test
    void 규격대로_이어붙여_해시하면_기대값과_같다() {
        // 전부 더미 값 (실제 키/상점ID 아님)
        String sig = sign("20260101000000", "testmid", "1000", "dummy-key");
        // 같은 입력이면 언제나 같은 서명 -> 결제사와 우리가 동일 결과를 내야 통과
        assertThat(sig).isEqualTo(sha256Hex("20260101000000testmid1000dummy-key"));
        assertThat(sig).hasSize(64);   // SHA-256 hex = 64자
    }

    @Test
    void 필드가_하나라도_바뀌면_서명이_달라진다() {
        String a = sign("20260101000000", "testmid", "1000", "dummy-key");
        String b = sign("20260101000000", "testmid", "1001", "dummy-key"); // 금액만 1원 차이
        assertThat(a).isNotEqualTo(b);   // 위·변조 감지의 근거
    }
}
```

## 소감
결제는 "돈이 오간다"는 이유로 막연히 어렵게 느껴졌는데, 막상 4단계로 분해하니 결국
*요청·확인의 반복*이었다. 어렵다고 겁먹기보다 흐름을 그림으로 그려놓고 하나씩 채우는 게
제일 빨랐다. 다음엔 취소/환불까지 붙여볼 예정.
