# 2026-07-03 법이 정한 계산을 코드로 옮기기

> 태그: #새기술 #결제 #도메인

## 무엇을 / 왜
구독 중도해지 환불을 붙였다. 그런데 환불 금액을 "우리 약관 문구"가 아니라 **법이 정한 기준**
(소비자분쟁해결기준)에 맞춰 계산해야 했다. 돈이 오가고 분쟁 소지가 큰 영역이라, 한 번 잘못
계산하면 고객 항의나 환불 사고로 이어진다. 정확도가 곧 신뢰였다.

## 어떻게 풀었나
법이 정한 계산을 그대로 코드로 옮겼다.
- **실제 결제한 금액 기준**으로 쓴 기간만큼 일 단위로 정산한다(할인 전 정가로 공제하는, 분쟁을
  부르는 방식은 배제).
- 하루라도 쓰면 그날은 사용한 것으로 보고(취소일 포함), **환불 하한은 0**(음수면 추가 청구 없이 0).
- 계산한 환불액을 결제사에 **원 거래 부분취소**로 걸어 실제 환불까지 연결했다.
- 같은 취소가 두 번 실행돼도 중복 환불이 안 되도록 **멱등**하게 설계했다.

가장 신경 쓴 건, 정책이 준 구체적인 예시(입력 -> 기대 환불액)를 먼저 테스트로 못박고 시작한 것이다.

## 💡 배운 것
- **돈 계산은 "문서 설명"이 아니라 "정책이 준 구체 예시"를 테스트로 고정하고 시작한다.** "절반 쓰면
  절반 환불" 같은 확정 사례를 먼저 코드로 박아두면, 이후 리팩터링해도 값이 흔들리지 않는다.
- **성공 케이스만큼 경계·하한을 명시한다.** 다 쓴 경우(환불 0), 취소일 포함 여부 같은 규칙을 빼먹으면
  음수 환불이나 하루치 오차가 난다.
- **외부 결제 취소 연동은 멱등하게.** 재시도·중복 호출이 있어도 두 번 환불되지 않게 설계해야 안전하다.

## 코드로 보기 (개념 재현 테스트)
법이 정한 일할 환불 공식을 더미 값으로 검증한다(실제 금액·거래 아님).

```java
// 목적: 중도해지 일할 환불 = 총결제 - 이용분 - 위약금(하한 0) 을 검증.
//  - 이용분 = 총결제 * 이용일수 / 총일수 (취소일 포함)
//  - base = 실제 결제한 금액(정가 아님)
class ProratedRefundTest {

    private long refund(long paid, int usedDays, int totalDays, long penalty) {
        long used = Math.round((double) paid * usedDays / totalDays);
        return Math.max(0, paid - used - penalty); // 하한 0: 음수면 추가청구 없이 0
    }

    @Test
    void 절반쯤_쓰면_절반쯤_환불() {
        // 3만원 / 30일 중 15일 사용 / 위약금 0 -> 15,000 환불
        assertThat(refund(30_000, 15, 30, 0)).isEqualTo(15_000);
    }

    @Test
    void 연간결제는_법정_한도내_위약금을_뗀다() {
        // 60만원 / 180일 중 60일 사용 / 위약금 10%(6만) -> 34만 환불
        assertThat(refund(600_000, 60, 180, 60_000)).isEqualTo(340_000);
    }

    @Test
    void 다_쓰면_환불은_음수가_아니라_0() {
        assertThat(refund(30_000, 30, 30, 0)).isZero();
    }
}
```

## 소감
"법대로 계산"이라는 말이 처음엔 막연했는데, 결국 입력과 기대 출력을 표로 정리해 테스트로 옮기는
일이었다. 규칙을 문장으로 읽는 것과 예시로 못박는 것은 확실히 달랐다. 돈을 다루는 로직일수록
"이 값이 맞다"를 코드가 스스로 증명하게 만들어 두니 마음이 편했다.
