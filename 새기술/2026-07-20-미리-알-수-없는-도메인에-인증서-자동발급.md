# 2026-07-20 미리 알 수 없는 도메인에도 인증서가 자동으로 나가게

> 태그: #새기술 #인프라 #HTTPS #멀티테넌트

## 무엇을 / 왜
전에 리버스 프록시를 앞에 세워 HTTPS 인증서를 자동으로 발급받게 만든 적이 있다. 그때는 설정 파일에
도메인을 적어두면 프록시가 알아서 인증서를 받아왔다. 도메인을 내가 미리 알고 있었으니 가능한 방식이었다.

그런데 이번엔 상황이 달랐다. 사용자가 자기 도메인을 사서 우리 서비스에 연결하는 구조가 되면서,
**어떤 도메인이 들어올지 미리 알 수 없게** 됐다. 지금 방식대로면 누가 도메인을 붙일 때마다 내가 서버
설정을 손으로 고쳐야 인증서가 나간다. 몇 명이면 몰라도 늘어나면 감당이 안 될 게 뻔했다.

## 어떻게 풀었나
찾아보니 "요청이 처음 들어온 순간에 인증서를 발급받는" 방식이 있었다. 미리 적어두지 않아도, 모르는
주소로 접속이 들어오면 그때 발급을 시도하는 것이다. 이거면 되겠다 싶었는데, 알아보니 그냥 켜두면
안 되는 거였다.

문제는 **누구나 자기 도메인이 우리 서버를 가리키게 만들 수 있다**는 점이다. 남의 서버 주소를 자기
도메인에 적어 넣는 걸 막을 방법이 없다. 그대로 열어두면 우리와 상관없는 도메인의 발급 요청이 계속
들어오고, 인증서 발급에는 하루 한도가 있어서 그걸 다 써버리면 정작 진짜 고객의 발급이 막힌다.

그래서 발급하기 직전에 한 번 되묻는 단계를 뒀다.

- 프록시가 모르는 주소를 만나면, 발급하기 전에 우리 서버에 "이 도메인 우리가 서비스하는 거 맞나?"를
  물어본다.
- 우리 쪽에선 등록되고 소유 확인까지 끝난 도메인일 때만 "맞다"고 답한다.
- "맞다"가 아니면 프록시는 발급을 하지 않는다.

등록된 도메인으로 접속하니 첫 요청에 인증서가 자동으로 나갔고, 등록하지 않은 아무 주소로 접속하니
연결이 거부됐다. 의도한 대로 됐다.

### 인증서만으로는 끝이 아니었다
여기까지 하고 "됐다" 싶었는데 화면이 여전히 오류였다. 인증서는 분명 나갔는데 페이지가 안 떴다.

원인은 다른 데 있었다. 들어온 주소를 보고 "어느 쇼핑몰인지" 찾아내는 부분이 기본으로 발급해준 주소만
보고 있었고, 사용자가 직접 붙인 도메인은 목록에 없으니 찾지 못했던 것이다. 그쪽도 같이 찾도록 고치고
나서야 화면이 제대로 떴다.

## 💡 배운 것
- 인증서 자동 발급에도 두 종류가 있다. 미리 아는 도메인을 적어두는 방식과, 모르는 도메인을 요청이
  들어온 순간 처리하는 방식. 서비스가 "사용자가 자기 도메인을 가져오는" 형태가 되면 후자가 필요해진다.
- 모르는 도메인에 자동 발급을 열 때는 **반드시 확인 단계를 같이 둬야 한다.** 남이 자기 도메인을 우리
  서버로 향하게 하는 걸 막을 수 없기 때문에, 열어두는 것만으로 발급 한도를 소진시키는 통로가 된다.
- "인증서가 나갔다"와 "서비스가 된다"는 다른 문제다. 연결이 암호화되는 것과, 그 주소로 어떤 내용을
  보여줄지 아는 것은 별개였다.
- 자동화를 만들어놓고 그 자리에서 끝내지 말고, 실제로 한 번 끝까지 태워봐야 빠진 게 보인다. 이번에도
  직접 접속해보지 않았으면 반쪽만 동작하는 채로 뒀을 것이다.

## 코드로 보기 (개념 재현 테스트)
발급 전 확인 단계가 왜 필요한지, 등록된 도메인만 통과시킨다는 게 어떤 동작인지 더미로 재현한다.

```java
// 목적: 모르는 도메인에 인증서를 자동 발급할 때, "우리가 서비스하는 도메인인가"를
//       먼저 확인해야 남용(발급 한도 소진)을 막을 수 있다.
class CertificateIssueGateTest {

    // 우리가 실제로 서비스하는 도메인 목록 (등록 + 소유 확인이 끝난 것만)
    static final Set<String> SERVED = Set.of("shop-a.example.com", "myshop.example.org");

    // 프록시가 발급 직전에 호출하는 확인 창구.
    // 200(true)이면 발급 진행, 아니면 발급 거부.
    static boolean mayIssueCertificate(String domain) {
        if (domain == null || domain.isBlank()) return false;
        // 대소문자/끝점 같은 표기 차이를 먼저 정리한 뒤 비교한다
        String normalized = domain.trim().toLowerCase().replaceAll("\\.$", "");
        return SERVED.contains(normalized);
    }

    @Test
    void 등록된_도메인만_발급을_허용한다() {
        // 등록된 도메인 - 표기가 달라도 통과해야 한다
        assertThat(mayIssueCertificate("shop-a.example.com")).isTrue();
        assertThat(mayIssueCertificate("SHOP-A.example.com.")).isTrue();

        // 우리와 무관한 도메인이 우리 서버를 가리키게 해둔 경우 - 반드시 거부
        assertThat(mayIssueCertificate("attacker.example.net")).isFalse();

        // 값이 비어 있으면 통과시키지 않는다 (열어두면 그대로 구멍이 된다)
        assertThat(mayIssueCertificate("")).isFalse();
        assertThat(mayIssueCertificate(null)).isFalse();
    }
}
```

## 소감
처음엔 "자동 발급 켜면 끝나는 거 아닌가" 하고 가볍게 봤는데, 열어두는 것 자체가 남용 통로가 될 수
있다는 걸 알고 조금 놀랐다. 편의 기능을 켤 때 "이걸 나쁘게 쓰면 어떻게 되지"를 같이 생각해야 한다는
걸 배웠다.

그리고 인증서까지 나간 걸 보고 다 됐다고 생각했다가 화면이 안 떠서 한 번 더 파고들었는데, 그 과정에서
"암호화 연결"과 "무엇을 보여줄지 아는 것"이 다른 층의 일이라는 게 몸으로 이해됐다. 만들고 나서 실제로
눌러보는 습관을 계속 들여야겠다.
