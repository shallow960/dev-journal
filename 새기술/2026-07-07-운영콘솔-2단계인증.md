# 2026-07-07 가장 센 권한 콘솔에 2단계 인증(TOTP) 붙이기

> 태그: #새기술 #인증 #보안

## 무엇을 / 왜
플랫폼 전체를 관리하는 운영자 콘솔은, 모든 고객사 데이터를 가로질러 볼 수 있는 가장 강한 권한을 가진다. 여기가 뚫리면 피해가 가장 크다. 그래서 일반 사용자 로그인보다 훨씬 강한 인증을 걸기로 했다 - 비밀번호에 더해 2단계 인증(2FA)을 요구하고, 접근 경로도 일반 사용자와 분리한다.

## 어떻게 풀었나
인증 앱(구글 OTP 등)이 보여주는 6자리 일회용 코드 방식(TOTP)을 붙였다. 원리는 의외로 단순하다 - 서버와 인증 앱이 같은 비밀 키를 공유하고, 양쪽이 "현재 시각(30초 단위)"과 그 키를 함께 해시해 같은 6자리 숫자를 만든다. 서버는 로그인 시 사용자가 입력한 코드가, 지금 시각으로 계산한 값과 맞는지 확인한다. 시계 오차를 감안해 앞뒤 30초 창까지는 허용한다. 비밀번호가 유출돼도, 그 순간의 인증 앱 코드까지 있어야만 로그인된다.

여기에 두 가지를 더했다 - (1) 인증 통과 후 발급되는 접근 토큰은 브라우저 스크립트가 읽을 수 없는 보안 쿠키에 담아, 화면단 스크립트 취약점으로 토큰이 새지 않게 했다. (2) 서버에 저장하는 공유 비밀 키는 평문이 아니라 암호화해서 보관했다.

## 💡 배운 것
- TOTP는 겉보기엔 복잡해 보여도 핵심은 "공유 비밀 키 + 현재 시각 -> 해시 -> 6자리"다. 외부 라이브러리 없이 표준 해시(HMAC-SHA1)만으로 서버와 앱이 같은 코드를 만들어낸다. 표준(RFC)이라 어떤 인증 앱과도 호환된다.
- 시계 오차 때문에 "정확히 지금 창"만 허용하면 정상 코드도 거부될 수 있다. 앞뒤 한 칸(±30초)까지 허용하는 게 실무 정석이다.
- 권한이 셀수록 인증을 강화하고 경로를 격리한다 - 가장 강한 콘솔은 2FA 필수 + 일반 로그인과 분리 + 토큰은 스크립트가 못 읽는 쿠키에. 보안은 "가장 위험한 곳"에 자원을 몰아주는 것.

## 코드로 보기 (개념 재현 테스트)
TOTP 코드 생성·검증의 핵심만 자기완결로 재현한다. (실제 서비스는 비밀 키를 암호화 저장하고, 검증만 서버가 한다.)

```java
// 목적: 인증 앱과 서버가 "같은 비밀 키 + 현재 시각"으로 동일한 6자리를 만들어내는 원리 재현.
class TotpTest {

    // 공유 비밀 키(실제로는 사용자마다 랜덤 생성 후 암호화 저장). 여기선 더미.
    private static final byte[] SECRET = "DUMMY-SHARED-SECRET".getBytes();
    private static final long STEP = 30; // 30초마다 코드가 바뀜

    // 현재 시각(초)을 30초 단위 카운터로 바꾸고, 키와 함께 해시해 6자리를 뽑는다.
    private String code(long epochSeconds) throws Exception {
        long counter = epochSeconds / STEP;
        Mac mac = Mac.getInstance("HmacSHA1");
        mac.init(new SecretKeySpec(SECRET, "HmacSHA1"));
        byte[] h = mac.doFinal(ByteBuffer.allocate(8).putLong(counter).array());
        int off = h[h.length - 1] & 0x0f;                    // 동적 오프셋(표준 truncation)
        int bin = ((h[off] & 0x7f) << 24) | ((h[off + 1] & 0xff) << 16)
                | ((h[off + 2] & 0xff) << 8) | (h[off + 3] & 0xff);
        return String.format("%06d", bin % 1_000_000);       // 6자리
    }

    // 서버 검증: 시계 오차 대비 앞뒤 한 칸(±30초)까지 허용
    private boolean verify(String input, long now) throws Exception {
        for (long off = -1; off <= 1; off++) {
            if (code(now + off * STEP).equals(input)) return true;
        }
        return false;
    }

    @Test
    void 같은_시각이면_앱과_서버가_같은_코드를_만든다() throws Exception {
        long now = 1_700_000_000L;
        assertThat(verify(code(now), now)).isTrue();
    }

    @Test
    void 창을_벗어난_옛_코드는_거부된다() throws Exception {
        long now = 1_700_000_000L;
        String oldCode = code(now - 120);       // 2분 전 코드
        assertThat(verify(oldCode, now)).isFalse();
    }
}
```

## 소감
2FA를 "그냥 라이브러리 쓰는 것"으로만 알았는데, 직접 원리를 재현해보니 "공유 키 + 시각 -> 해시"라는 뼈대가 생각보다 단순했다. 그리고 보안은 모든 문을 똑같이 조이는 게 아니라, 제일 위험한 문(가장 센 권한)에 자원을 몰아주는 거라는 감을 이번에 잡았다.
