# 2026-07-03 신원과 권한 범위를 나눈 2단계 토큰

> 태그: #새기술 #인증 #설계

## 무엇을 / 왜
한 계정이 여러 조직(각각 독립된 데이터·권한)을 넘나들며 관리하는 구조를 붙였다. 이때 "이 사람이
누구냐"와 "지금 어느 조직을 다루는 중이냐"를 로그인 토큰 하나에 다 담으려니 애매했다. 한 토큰에
특정 조직을 박아버리면 조직을 바꿀 때마다 재로그인해야 하고, 아무 조직이나 담으면 그 토큰이
유출됐을 때 전 조직이 노출된다.

## 어떻게 풀었나
토큰을 **2단계**로 나눴다.
- **계정 토큰**: 로그인하면 받는, "누구인지"만 담은 토큰. 특정 조직에 묶이지 않는다. 내 조직 목록
  보기, 조직 선택에만 쓴다.
- **스코프 토큰**: 특정 조직을 고르면, 그 조직 소속이 맞는지 검증한 뒤 **그 조직에만 유효한 짧은
  수명의 토큰**을 발급한다. 실제 데이터 조작은 이걸로 한다.

스코프 토큰이 만료되면 화면이 계정 토큰으로 조용히 새로 발급받는다(사용자는 모른다). 다른 조직으로
넘어갈 땐 계정 토큰으로 그 조직의 스코프 토큰을 다시 받으면 된다.

## 💡 배운 것
- **"누구냐"와 "무엇에 대한 권한이냐"를 분리하면**, 로그인 하나로 여러 스코프를 안전하게 오갈 수 있다.
  각 스코프 토큰은 딱 한 조직의 권한만 가져 격리가 유지된다.
- **스코프 토큰은 짧게 + 자동 재발급**이 핵심이다. 짧아서 유출 피해가 작고, 자동 재발급이라 작업이
  안 끊긴다. 만료 갱신 책임을 수명 긴 토큰(계정) 하나에만 둬서 단순해졌다.
- **기존 발급 로직 위에 클레임 두 개(계정 식별자, 평면 구분)만 얹어** 최소 변경으로 확장했다.
  단일 조직만 쓰던 기존 로그인 흐름은 그대로 둔 채.

## 코드로 보기 (구조 골격)
계정 토큰과 스코프 토큰의 역할 분리, 그리고 스코프 토큰이 소속 검증을 거쳐 발급되는 골격을 재현한다.

```java
// 목적: 계정 토큰(신원) / 스코프 토큰(특정 조직 권한)의 분리와 "선택 시 검증 후 발급"을 재현.
class TwoTierTokenTest {

    private static final long TTL_SECONDS = 15 * 60; // 스코프 토큰은 짧게

    record AccountToken(String accountId) {}                                   // 누구인지만
    record ScopeToken(String accountId, String orgId, String role, long exp) {} // 한 조직에 한정

    // 저장된 소속(누가 어느 조직에서 무슨 역할인지)
    private String membershipRole(String accountId, String orgId) {
        if (accountId.equals("acc1") && orgId.equals("orgA")) return "owner";
        if (accountId.equals("acc1") && orgId.equals("orgB")) return "staff";
        throw new SecurityException("not a member");
    }

    // 조직 선택 = 계정 토큰 + 소속 검증 -> 그 조직 스코프 토큰 발급
    private ScopeToken openOrg(AccountToken account, String orgId, long now) {
        String role = membershipRole(account.accountId(), orgId); // 소속 아니면 거부
        return new ScopeToken(account.accountId(), orgId, role, now + TTL_SECONDS);
    }

    @Test
    void 같은_계정이_조직마다_다른_역할로_전환된다() {
        AccountToken account = new AccountToken("acc1");
        assertThat(openOrg(account, "orgA", 0).role()).isEqualTo("owner"); // A 에선 오너
        assertThat(openOrg(account, "orgB", 0).role()).isEqualTo("staff"); // B 에선 스태프
    }

    @Test
    void 소속_아닌_조직은_스코프토큰이_발급되지_않는다() {
        assertThatThrownBy(() -> openOrg(new AccountToken("acc1"), "orgZ", 0))
            .isInstanceOf(SecurityException.class);
    }
}
```

## 소감
처음엔 "토큰에 조직 목록을 다 넣으면 되지 않나" 싶었는데, 유출 반경을 생각하니 답이 아니었다.
"넓은 신원 + 좁은 권한"으로 나누는 순간 전환 UX도 격리도 같이 풀렸다. 권한은 늘 필요한 만큼만
좁게 주는 게 결국 단순하고 안전하다는 걸 몸으로 느꼈다.
