# 2026-07-15 비밀 키는 암호화해 저장하고 화면엔 일부만 보인다

> 태그: #새기술 #보안 #암호화

## 무엇을 / 왜
쇼핑몰 주인이 결제사 연동 키나 소셜 로그인 비밀 키를 입력하는 설정 화면을 만들었다. 처음엔 입력받은
값을 그대로 데이터베이스에 저장하려고 했다. 저장했다가 다시 불러와 화면에 보여주면 되니까.

## 어떻게 풀었나
그런데 이런 키는 한 번 새어 나가면 곧바로 악용되는 값이라, 데이터베이스에 그냥(평문) 두면 안 된다는 걸
알게 됐다. 두 가지를 나눠서 처리했다.
- 저장할 때: 값을 암호화해서 넣는다. 데이터베이스만 봐서는 원문을 알 수 없다.
- 화면에 보여줄 때: 전체를 다시 노출하지 않고, 앞 몇 글자만 남기고 나머지는 가리는 마스킹으로 보여준다.
  "값이 등록돼 있다"는 건 확인되지만 원문은 안 드러난다.

즉 원문은 저장하는 순간에만 잠깐 쓰이고, 그 뒤로는 어디에서도 통째로 나타나지 않게 했다. 결제 설정과
소셜 로그인 설정 두 화면에 같은 방식을 똑같이 적용했다.

## 💡 배운 것
- 민감한 값은 "저장은 암호화, 표시는 마스킹"이 한 세트다. 둘 중 하나만 하면 반쪽이다.
- 화면에 원문을 다시 뿌리지 않아도 사용자는 "등록됨"만 확인하면 충분하다. 편의를 위해 원문을 재노출하는
  건 위험을 부른다.
- 같은 성격의 값(키·시크릿)은 도메인이 달라도 같은 취급 규칙을 재사용하면 일관돼서 실수가 준다.

## 코드로 보기 (개념 재현 테스트)
원문은 저장할 때만 쓰이고, 화면엔 마스킹된 형태만 나가는 흐름을 더미로 재현한다.

```java
// 목적: 저장은 암호문, 표시는 일부만 - 원문이 화면에 통째로 나가지 않음을 보인다.
class SecretHandlingTest {

    // 개념용 더미 암복호화 (실제로는 키 관리되는 표준 암호화를 쓴다)
    static String encrypt(String raw) { return "enc(" + raw + ")"; }
    static String decrypt(String enc) { return enc.substring(4, enc.length() - 1); }

    static String onSave(String rawSecret) {        // 저장: 암호화해 보관
        return encrypt(rawSecret);
    }

    static String forDisplay(String stored) {       // 표시: 앞 4글자만 남기고 가림
        String raw = decrypt(stored);
        int keep = Math.min(4, raw.length());
        return raw.substring(0, keep) + "*".repeat(raw.length() - keep);
    }

    @Test
    void 저장은_암호문_표시는_마스킹() {
        String stored = onSave("sk_live_ABCDEFGH");
        assertThat(stored).doesNotContain("ABCDEFGH");                 // DB엔 원문이 없음
        assertThat(forDisplay(stored)).isEqualTo("sk_l************");  // 화면엔 일부만
    }
}
```

## 소감
그냥 저장하면 훨씬 편한데, 그 편함이 곧 위험이라는 걸 배웠다. 특히 "화면에서 다시 보여줘야 하니 원문이
필요하지 않을까?" 싶었지만, 사용자는 "등록됐다"는 사실만 알면 되지 원문을 다시 볼 필요가 없다는 걸 깨닫고
나니 마스킹이 당연하게 느껴졌다. 민감한 값일수록 한 겹 더 감싸는 게 기본이라는 감각이 생겼다.
