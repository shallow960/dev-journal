# 2026-07-14 행 수준 보안으로 여러 세입자의 데이터를 쓰기까지 격리

> 태그: #새기술 #보안 #멀티테넌트

## 무엇을 / 왜
한 플랫폼 위에 여러 세입자(입점 스토어)가 각자 데이터를 갖는 구조에서, 관리 화면 여러 개를 실제
데이터로 연결했다. 절대 조건은 "한 세입자의 화면에서 다른 세입자의 데이터가 보이거나 수정되면 안 된다"였다.
이걸 매 조회문마다 "내 것만" 조건을 손으로 붙여서 지키면, 화면이 늘수록 한 군데만 빠뜨려도 사고가 난다.

## 어떻게 풀었나
세입자 구분을 개별 쿼리의 책임으로 두지 않고, 데이터베이스의 행 수준 보안으로 밑단에서 강제했다.
- 관리 화면에 들어올 때 토큰에서 현재 세입자를 확정하고, 그 범위를 요청 세션에 건다.
- 이후 모든 조회·수정은 자동으로 그 세입자의 행으로만 한정된다. 쿼리에 조건을 안 붙여도 된다.
- 수정까지 안전하다. 다른 세입자의 항목을 겨냥한 수정 요청이 와도, 그 행은 애초에 범위 밖이라
  수정 대상이 0건이 되고 "없음"으로 처리된다. 코드에서 조건을 빠뜨려도 교차 접근이 불가능하다.

덕분에 조회 화면 여러 개와 관리(쓰기) 기능을 한 가지 패턴으로 반복하면서도, 세입자 격리는 매번
신경 쓰지 않아도 밑단에서 보장됐다.

## 💡 배운 것
- 멀티테넌시 격리는 "매 쿼리 조건"이 아니라 "밑단 기본값"으로 세운다. 사람이 빠뜨릴 수 있는 방어는
  언젠가 빠뜨린다. 규칙을 데이터 계층에 박아두면 실수해도 안전하다.
- 읽기만큼 쓰기도 격리한다. 조회만 막고 수정은 열려 있으면 반쪽이다. 범위 밖 대상은 "수정 0건 =
  없음"으로 귀결되게 하면, 잘못된 대상 수정이 조용히 통과하지 않는다.
- 밑단에서 격리가 보장되면 화면별 구현이 단순·일관해져 생산성도 같이 오른다.

## 코드로 보기 (개념 재현 테스트)
범위(현재 세입자)를 세션에 걸면 다른 세입자의 행이 읽기·쓰기에서 사라지는 걸 더미로 재현한다.
실제로는 데이터베이스의 행 수준 보안이 이 역할을 한다.

```java
// 목적: "현재 세입자" 범위를 세션에 걸면, 남의 행은 조회/수정 대상에서 자동 제외됨을 보인다.
class RowScopedAccessTest {

    record Row(String tenant, String id, boolean visible) {}

    // 현재 세입자 범위로만 보이게 거르는 뷰 (DB의 행 수준 보안 흉내)
    static class ScopedTable {
        private final List<Row> rows;
        private String currentTenant; // 세션에 걸린 범위

        ScopedTable(List<Row> rows) { this.rows = new ArrayList<>(rows); }
        void useTenant(String t) { this.currentTenant = t; }

        List<Row> find() { // 범위 밖은 애초에 안 보임
            return rows.stream().filter(r -> r.tenant().equals(currentTenant)).toList();
        }

        int setVisible(String id, boolean v) { // 범위 밖은 대상 0건
            var hit = rows.stream()
                .filter(r -> r.tenant().equals(currentTenant) && r.id().equals(id)).toList();
            hit.forEach(r -> rows.set(rows.indexOf(r), new Row(r.tenant(), r.id(), v)));
            return hit.size(); // 0 이면 "없음" 으로 귀결
        }
    }

    @Test
    void 남의_세입자_행은_조회에도_수정에도_잡히지_않는다() {
        var table = new ScopedTable(List.of(
            new Row("A", "x", true),
            new Row("B", "y", true)));

        table.useTenant("A");
        assertThat(table.find()).extracting(Row::id).containsExactly("x"); // A 것만 보임

        int affected = table.setVisible("y", false); // B의 행을 수정 시도
        assertThat(affected).isZero(); // 범위 밖 -> 0건 -> "없음", 교차 수정 불가
    }
}
```

## 소감
"조건을 잘 붙이자"는 다짐은 화면이 스무 개쯤 되면 반드시 한 번은 배신당한다. 그래서 방어를 사람의
주의력에서 데이터 계층으로 내려놓으니, 새 화면을 붙일 때 격리를 아예 생각하지 않아도 되는 게 편했다.
읽기만 막다가 쓰기에서 새는 사고가 흔한데, "범위 밖은 수정 0건"이라는 성질 하나로 그 구멍까지
닫힌 게 특히 든든했다.
