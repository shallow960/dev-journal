# 2026-07-03 한 트랜잭션에서 다른 조직으로 안전하게 쓰기

> 태그: #새기술 #멀티테넌시 #데이터베이스

## 무엇을 / 왜
여러 조직이 한 데이터베이스를 공유하는 구조에서, **행별 격리(Row-Level Security)** 를 쓴다.
각 요청은 "지금 어느 조직인지"를 세션 변수에 담고, DB는 그 값과 일치하는 행만 보이고 쓸 수 있게
막는다. 그런데 한 요청(= 한 트랜잭션) 안에서 **새 조직을 만들고, 그 조직의 데이터를 곧바로 넣어야**
하는 상황이 생겼다. 문제는 트랜잭션이 시작될 때 세션의 조직 값이 이미 (다른 조직이거나 비어 있게)
정해져 있어서, 새 조직으로 넣으려는 데이터가 격리 규칙에 걸려 **조용히 0행 처리**됐다 - 에러도 안 났다.

## 어떻게 풀었나
새 조직을 만든 직후, **그 트랜잭션의 세션 조직 값을 새 조직으로 명시적으로 바꿔** 줬다.
- 요청 진입 시 자동으로 세팅되는 값과 별개로, 트랜잭션 도중에는 손으로 다시 설정해야 한다.
- 이 전환을 트랜잭션 로컬로 걸어, 그 트랜잭션이 끝나면 자연히 원복되게 했다.
- 이후 인서트들은 새 조직 격리를 정상 통과했다.

조직 경계를 넘는 쓰기는 위험하므로, 이 전환은 "새 조직 프로비저닝" 같은 **명확히 의도된 지점에만
국소적으로** 뒀다.

## 💡 배운 것
- **행별 격리는 "현재 세션/트랜잭션의 컨텍스트"에 전적으로 의존한다.** 트랜잭션 도중 대상 조직을
  바꾸려면 그 컨텍스트를 반드시 갱신해야 한다 - 안 하면 막힌 게 아니라 **말없이 0행**이라 더 위험하다.
- **자동 세팅(요청 진입)과 수동 전환(트랜잭션 중간)은 다른 문제다.** 프레임워크가 넣어주는 값에만
  기대면, 흐름 중간에 대상이 바뀌는 케이스를 놓친다.
- **경계를 넘는 쓰기는 좁게.** 전환은 꼭 필요한 한 지점에만 두고, 끝나면 원래 컨텍스트로 돌아오게 했다.

## 코드로 보기 (구조 골격)
새 조직을 만든 뒤, 트랜잭션의 조직 컨텍스트를 전환하고 그 조직 데이터를 넣는 골격이다.

```java
// 목적: 한 트랜잭션에서 새 조직 생성 -> 컨텍스트 전환 -> 그 조직 데이터 인서트.
@Transactional
public UUID provisionOrg(String name) {
    UUID orgId = UUID.randomUUID();

    // 1) 조직 레코드 자체는 격리 대상이 아닌 전역 테이블에 먼저 만든다.
    globalOrgTable.insert(orgId, name);

    // 2) 이 트랜잭션의 "현재 조직" 세션 변수를 새 조직으로 전환(트랜잭션 로컬).
    //    이 한 줄이 없으면 아래 인서트가 격리에 걸려 조용히 0행이 된다.
    db.exec("select set_current_org(?, /* transaction-local */ true)", orgId);

    // 3) 이제 격리 대상 테이블에도 새 조직 데이터가 들어간다.
    memberTable.insert(orgId, /* ...owner... */);
    settingsTable.insert(orgId, /* ...defaults... */);

    return orgId; // 트랜잭션 종료 시 세션 컨텍스트는 자연히 원복
}
```

## 소감
가장 애먹은 건 "에러가 안 났다"는 점이었다. 막혔으면 예외라도 났을 텐데, 격리는 조용히 0행으로
지나가서 한참 헤맸다. 격리를 쓰는 코드에선 "이 쿼리가 보는 조직이 지금 뭐지"를 늘 의식하게 됐다.
편리한 자동화(요청 단위 세팅) 뒤에 숨은 전제를, 흐름이 그 전제를 벗어나는 순간 직접 챙겨야 한다.
