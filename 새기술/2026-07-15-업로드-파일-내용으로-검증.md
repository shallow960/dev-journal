# 2026-07-15 업로드 파일은 확장자가 아니라 내용으로 검증한다

> 태그: #새기술 #보안 #파일업로드

## 무엇을 / 왜
이미지 업로드 기능을 만들다가 문득 "이게 진짜 이미지인 걸 어떻게 확신하지?"라는 생각이 들었다.
처음엔 파일 이름 끝의 확장자(.jpg)나, 브라우저가 "이건 이미지예요"라고 알려주는 타입 정보를 보면
되는 줄 알았다.

## 어떻게 풀었나
찾아보니 그 값들은 사용자가 얼마든지 바꿀 수 있는 "자기 주장"이라 믿으면 안 됐다. 실행 파일 이름만
.jpg로 바꿔 올리면 검사를 통과해버린다. 그래서 이름이 아니라 파일이 실제로 담고 있는 내용, 그중에서도
맨 앞 몇 바이트(형식마다 정해진 고정된 서명)를 열어서 형식을 판별하도록 했다.
- 업로드 처리를 한 군데로 모았다.
- 들어온 파일의 앞부분 바이트를 읽어, 허용한 형식(이미지 등)의 서명과 맞는지 확인한다.
- 통과한 파일만 공개 폴더와 분리된 곳에 저장하고, 되돌려줄 때도 안전한 경로로만 내보낸다.
- 새 상품 이미지 업로드에 먼저 붙이고, 예전에 만들어둔 문의 첨부 기능도 같은 검사로 고쳤다.

## 💡 배운 것
- 파일 형식은 이름이 아니라 내용으로 판단한다. 확장자·선언된 타입은 사용자가 바꿀 수 있어 신뢰 근거가
  못 된다.
- "사용자가 준 정보는 일단 의심한다"가 보안의 출발점이라는 걸 몸으로 이해했다.
- 검증만으로 끝이 아니라 저장 위치(공개 폴더와 분리)·되돌려주는 경로까지 한 세트로 챙겨야 안전하다.

## 코드로 보기 (개념 재현 테스트)
이름(확장자)은 속아도 내용(앞부분 서명 바이트)은 못 속인다는 걸 더미로 재현한다.

```java
// 목적: 확장자만 믿으면 위장 파일이 통과하지만, 앞부분 서명 바이트로 검사하면 걸러진다.
class FileTypeByContentTest {

    // 형식별 "고정된 앞부분 서명" (매직 바이트) - 값은 예시
    static final byte[] PNG_SIG = {(byte) 0x89, 'P', 'N', 'G'};

    static boolean looksLikePng(byte[] bytes) { // 이름이 아니라 내용으로 판단
        if (bytes.length < PNG_SIG.length) return false;
        for (int i = 0; i < PNG_SIG.length; i++) {
            if (bytes[i] != PNG_SIG[i]) return false;
        }
        return true;
    }

    @Test
    void 확장자는_속여도_내용_검사는_못_속인다() {
        // 실행 파일 내용인데 이름만 image.png 로 위장
        byte[] disguised = {'M', 'Z', 0x00, 0x00}; // 실행 파일의 앞부분 서명(예시)
        String fakeName = "image.png";

        assertThat(fakeName.endsWith(".png")).isTrue();   // 이름만 보면 통과 - 위험
        assertThat(looksLikePng(disguised)).isFalse();    // 내용으로 보면 거른다 - 안전
    }
}
```

## 소감
"확장자만 보면 되겠지"가 얼마나 순진한 가정이었는지 알게 됐다. 겉보기엔 잘 돌아가는 업로드 기능이라도
"이걸 악용하려면 어떻게 할까?"를 한 번 떠올려보는 습관이 필요하다는 걸 배웠다. 새 기능뿐 아니라 예전에
대충 만들어둔 첨부 기능까지 같이 고치면서, 한 번 세운 안전 기준은 옛 코드에도 소급해야 진짜 지켜진다는
것도 느꼈다.
