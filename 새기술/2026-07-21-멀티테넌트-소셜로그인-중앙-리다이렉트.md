# 2026-07-21 하위 도메인마다 다른 가게에 소셜 로그인을 한 번에 붙이기

> 태그: #새기술 #멀티테넌시 #OAuth #인증

## 무엇을 / 왜
한 플랫폼이 여러 가게(쇼핑몰)를 만들어주고, 가게마다 주소가 다르다 - `가게이름.플랫폼도메인` 식의
하위 도메인이다. 이 각각의 가게에 카카오·네이버·구글 같은 소셜 로그인을 붙이려고 했다.

문제는 소셜 로그인의 동작 방식이었다. 이런 로그인은 "**미리 등록해 둔 주소로만** 인증을 끝내고
되돌아오게" 해준다. 그런데 가게가 생길 때마다 주소가 새로 늘어나는데, "우리 도메인 하위 전체를
허용"하는 와일드카드를 받아주지 않는다. 그대로라면 가게가 하나 늘 때마다 소셜 로그인 콘솔에 주소를
일일이 등록해야 한다 - 자동화가 안 되는 구조다.

## 어떻게 풀었나
**대표 도메인 한 곳**만 소셜 로그인의 "되돌아올 주소"로 등록하고, 모든 가게가 그 대표 도메인을 거쳐
인증받은 뒤 원래 가게로 돌아오게 했다. 흐름은 이렇다.

1. 손님이 A가게에서 로그인 버튼을 누른다 → 서버가 "지금 A가게에서 시작했다"는 정보를 임시 상태값에
   저장하고, 소셜 로그인으로 보낸다. 이때 되돌아올 주소는 **대표 도메인**으로 지정한다.
2. 소셜 로그인이 끝나면 대표 도메인 콜백으로 돌아온다.
3. 대표 도메인은 상태값을 꺼내 "원래 A가게였음"을 복원하고, A가게 소속으로 로그인 처리를 한다.
4. 1회용 인계 코드를 발급해 A가게로 되돌려보낸다. A가게가 그 코드를 교환해 **자기 도메인에** 세션을 세운다.

여기서 함정이 여럿이었다.

- **콜백은 "어느 가게인지"를 모른다.** 대표 도메인은 특정 가게가 아니므로, 원래 가게 정보를 상태값에
  실어 보내 복원해야 한다. 이 상태값은 추측 불가한 임의 토큰 + 짧은 만료 + 1회성으로 보호했다.
- **되돌려보낼 주소는 반드시 검증.** 상태값에 담긴 복귀 주소가 우리 도메인 소속이 맞는지 확인하지
  않으면, 조작해서 엉뚱한 사이트로 되돌려보내는 공격(오픈 리다이렉트)에 열린다.
- **쿠키는 도메인을 넘지 못한다.** 대표 도메인에서 세운 로그인 세션(쿠키)은 주소가 다른 A가게로 그냥
  넘어가지 않는다. 그래서 대표 도메인은 세션을 직접 넘기는 대신 **1회용 인계 코드**만 넘기고, A가게가
  그 코드를 교환해 자기 도메인에 세션을 새로 세우게 했다.
- **대표 도메인 경로는 "어느 가게 데이터인지"가 자동으로 안 걸린다.** 로그인 처리(사용자 조회·생성)가
  가게별 데이터 격리를 통과하려면, 그 가게의 컨텍스트를 콜백 안에서 손으로 걸어줘야 했다.

실제 소셜 로그인 키를 꽂기 전에도 이 왕복 전체(시작 → 대표 도메인 콜백 → 원래 가게 복귀 → 세션 확립)가
처음부터 끝까지 도는 것을 확인해 뒀다. 키만 넣으면 켜지는 상태다.

## 💡 배운 것
- **외부 서비스의 제약이 내 아키텍처를 바꾼다.** "로그인 버튼 붙이기"로 보였는데, 와일드카드 미지원이라는
  제약 하나 때문에 대표 도메인 경유 구조로 크게 돌아가야 했다. 붙이기 전에 제약부터 확인해야 한다.
- **도메인이 다르면 쿠키가 안 넘어간다.** 크로스 도메인 세션은 "대표에서 세우고 넘기기"가 아니라
  "**1회용 코드를 넘기고 목적지에서 재발급**"이 정석이다.
- **되돌아올 주소는 항상 화이트리스트 검증.** 사용자가 준(또는 상태값에 실린) 복귀 주소를 그대로 믿으면
  오픈 리다이렉트가 된다.
- **테넌트가 자동 해석되지 않는 경로에서 데이터 격리를 쓰려면 컨텍스트를 손으로 걸어야 한다** -
  안 걸면 막히는 게 아니라 조용히 0행/거부라 더 헷갈린다.

## 코드로 보기 (구조 골격)
가게에서 로그인을 시작하고(상태 저장), 대표 도메인 콜백이 원래 가게로 복원해 인계 코드로 되돌려보내는
골격이다. 실제 키·도메인·내부 이름은 뺐고 값은 더미다.

```java
// [1] 가게 도메인: 로그인 시작 - "어느 가게에서 시작했나"를 상태값에 저장하고 대표 도메인으로 보낸다.
String startLogin(String provider, String originHost, String tenantId) {
    // 되돌려보낼 곳은 우리 가게 도메인만 허용(오픈 리다이렉트 차단).
    if (!isOurStoreDomain(originHost)) throw new BadRequest("origin not allowed");

    String state = randomToken();                       // 추측 불가 + 짧은 만료 + 1회성
    stateStore.save(state, tenantId, originHost, /* ttl */ Duration.ofMinutes(10));

    // 소셜 로그인의 redirect_uri = 대표 도메인 콜백 (와일드카드 대신 대표 한 곳만 등록)
    return authorizeUrl(provider)
        + "?redirect_uri=" + enc("https://auth.example.com/oauth/callback/" + provider)
        + "&state=" + enc(state);
}

// [2] 대표 도메인 콜백: 상태로 원래 가게 복원 -> 그 가게 컨텍스트로 로그인 -> 1회용 인계 코드로 되돌림.
String centralCallback(String provider, String code, String state) {
    var s = stateStore.consume(state);                  // 없거나 만료면 실패(1회성)
    if (s == null) throw new BadRequest("invalid state");

    Profile profile = verifyWithProvider(provider, code); // (키 붙기 전엔 목으로 대체)

    // 대표 도메인은 테넌트가 자동 해석 안 됨 -> 그 가게 데이터 격리를 "손으로" 건다.
    withTenantContext(s.tenantId(), () -> {
        User user = findOrCreate(profile);
        Session session = issueSession(user);           // 여기서 만든 세션 쿠키는 이 도메인 것
        String handoff = randomToken();                 // 도메인을 못 넘으니 1회용 코드만 넘긴다
        handoffStore.save(handoff, s.tenantId(), session, Duration.ofMinutes(2));
        redirect("https://" + s.originHost() + "/auth/handoff?code=" + handoff);
    });
}

// [3] 원래 가게 도메인: 인계 코드를 교환해 "이 도메인에" 세션을 세운다(교환은 원래 가게에서만 허용).
Session exchangeHandoff(String handoffCode, String currentTenant) {
    var h = handoffStore.consume(handoffCode);
    if (h == null || !h.tenantId().equals(currentTenant)) throw new Forbidden();
    return h.session(); // 이 응답에서 이 도메인용 로그인 쿠키를 세팅
}
```

## 소감
겉보기엔 로그인 버튼 하나였는데, "가게마다 주소가 다르다 + 외부 서비스가 와일드카드를 안 받는다"는
두 사실이 만나 구조를 크게 바꿔야 했다. 제일 오래 붙든 건 "왜 로그인은 됐는데 원래 가게에선 안
로그인 상태지?"였고, 답은 쿠키가 도메인을 못 넘는다는 기본기였다. 화려한 기능보다 이런 기본 제약을
먼저 아는 게 설계를 좌우한다는 걸 다시 느꼈다. 그리고 되돌아올 주소는 무조건 검증한다 - 편의를 위한
리다이렉트가 그대로 공격 통로가 될 수 있다.
