# 2026-07-23 "관리자냐"만 물었지 "어느 가게의 관리자냐"는 안 물었다

> 태그: #트러블슈팅 #인증 #인가 #멀티테넌시 #보안

## 무엇을 / 왜
한 코드로 여러 가게(각각 독립된 데이터)를 운영하는 서비스에서, 각 가게의 관리자 화면이 어떻게
보호되는지 점검하다가 큰 구멍을 발견했다.

관리자 화면은 요청이 올 때 **"당신이 관리자입니까?"(역할)만 확인**하고 있었다. 그런데 **정작 어느
가게를 관리하는지는 요청이 실어 보낸 웹 주소(가게 주소)로 결정**하고 있었다. 문제는 이 서비스에서
**모든 가게 주인이 자기 가게의 "관리자"** 라는 점이다. 이 둘을 합치면:

- A 가게 주인이 자기 로그인은 그대로 둔 채, **주소만 B 가게로 바꿔** 요청을 보낸다.
- 서버는 "관리자 맞네(역할 OK)" -> "요청 주소가 B니까 B 가게로 처리" 를 한다.
- 결과: **남의 가게(B)의 주문 - 회원 정보 - 상품을 마음대로 조작**할 수 있다.

데이터가 가게별로 새지 않게 막아주는 데이터베이스 차원의 격리(행 수준 보안)도 있었지만, 그 격리
역시 "어느 가게냐"를 **같은 웹 주소로 판단**했다. 그래서 공격자가 주소를 B로 고르는 순간 격리도
B를 향하게 되어, 이 경우엔 방어가 되지 않았다. 같은 패턴이 상품 편집 쪽에도 있어서, 아무 판매자나
주소만 바꾸면 남의 가게 상품을 고치거나 지울 수 있었다.

## 어떻게 풀었나
핵심은 **역할 확인이 "관리자입니까"에는 답하지만 "어느 가게의 관리자입니까"에는 답하지 못한다**는
것을 깨달은 것이다.

로그인 토큰에는 이미 **그 사람이 소속된 가게**가 새겨져 있었다. 그래서 규칙을 하나 추가했다.

- **토큰에 새겨진 소속 가게 == 요청이 처리하려는 가게(주소로 정해진)** 를 반드시 맞춰본다.
- 다르면 거부한다. A 소속 로그인으로 B를 처리하려 하면 여기서 막힌다.

그리고 이걸 화면마다 하나씩 막지 않았다. 새 화면이 생길 때마다 또 뚫릴 테니까. **모든 요청이 지나는
공통 관문 한 곳**에서 이 확인을 하게 해서, 관리자 화면도 상품 편집도 앞으로 생길 화면도 **한 번에**
같은 규칙으로 닫히게 했다.

관문은 **토큰에 소속 가게가 실려 있을 때만** 이 검사를 한다. 로그인 안 한 공개 조회나, 애초에 특정
가게에 소속되지 않은 토큰은 그냥 통과시켜 다른 관문에 맡긴다. 공격에 쓸 수 있는 토큰(역할을 가진)은
반드시 소속 가게가 실려 있으므로, 이 관대함이 우회로가 되지는 않는다.

A 가게 로그인으로 B 가게에 접근을 시도하면 차단되는 것, 자기 가게 접근은 정상인 것을 테스트로
확인하고 전체 검사를 통과시킨 뒤 배포했다.

## 💡 배운 것
- **"누구냐(역할)"와 "어느 조직 소속이냐"는 다른 질문이다.** 여러 조직을 한 코드로 운영하면 역할만
  맞아선 안 되고, 그 사람이 **지금 건드리는 조직에 속하는지**까지 봐야 한다. 역할 통과 = 인가 완료가
  아니다.
- **데이터 격리도 "어느 조직이냐"의 기준점이 공격자가 고를 수 있는 값이면 소용없다.** 격리가 걸려
  있어도 그 기준(여기선 웹 주소)을 요청자가 정하면, 격리는 공격자가 지목한 조직을 향한다. 기준점
  자체가 **신뢰할 수 있는 출처(서명된 토큰)** 여야 한다.
- **같은 부류의 결함은 화면마다가 아니라 공통 관문에서 한 번에 닫는다.** 하나씩 막으면 다음에 생기는
  화면에서 또 뚫린다. 근본 위치에 규칙을 두면 미래의 화면까지 함께 보호된다.

## 코드로 보기 (개념 재현 테스트)
역할만 확인하면 주소만 바꿔 남의 조직을 조작할 수 있고, "소속 조직 == 대상 조직"을 함께 강제하면
막히는 것을 재현한다. 실제 내부 이름은 뺐고 값은 더미다.

```java
// 목적: "관리자냐"(역할)만 확인하고 대상 조직을 요청 주소로 정하면,
//       로그인을 그대로 둔 채 주소만 바꿔 남의 조직을 조작할 수 있음을 재현.
//       로그인에 새겨진 "소속 조직" == "대상 조직" 을 강제하면 막힌다.
class TenantBindingTest {

    // 요청 하나를 흉내낸다: 로그인 토큰(역할 + 소속 조직) + 대상 조직(요청 주소로 결정).
    record Request(String role, String tokenTenant, String hostTenant) {}

    // (취약) 역할만 본다. 대상 조직은 주소(hostTenant)가 정하고, 데이터 격리도 그 값에 걸린다.
    static boolean allowedByRoleOnly(Request r) {
        return "ADMIN".equals(r.role());   // "관리자냐?" 에만 답한다
    }

    // (수정) 역할 + "로그인에 새겨진 소속 조직 == 대상 조직" 을 함께 본다.
    static boolean allowedWithBinding(Request r) {
        if (!"ADMIN".equals(r.role())) return false;
        return r.tokenTenant() != null && r.tokenTenant().equals(r.hostTenant());
    }

    @Test
    void 역할만_보면_주소만_바꿔_남의_조직을_조작할_수_있다() {
        // A 조직 관리자가 로그인은 그대로, 주소만 B 로 바꿔 요청한다.
        Request crossTenant = new Request("ADMIN", "orgA", "orgB");
        assertThat(allowedByRoleOnly(crossTenant)).isTrue();   // 통과되어 버린다(취약)
    }

    @Test
    void 소속과_대상을_맞추면_크로스_조직_접근이_막힌다() {
        assertThat(allowedWithBinding(new Request("ADMIN", "orgA", "orgB"))).isFalse(); // 남의 조직 차단
        assertThat(allowedWithBinding(new Request("ADMIN", "orgA", "orgA"))).isTrue();  // 자기 조직은 정상
    }
}
```

## 소감
"관리자만 들어오게 막았으니 됐지" 싶었는데, 그 확인은 문제의 절반("관리자냐")에만 답하고 있었다.
나머지 절반("어느 가게의 관리자냐")을 빼먹은 게 남의 가게를 통째로 넘길 수 있는 구멍이 됐다. 예전에
결제 화면에서 "대상은 주소가 아니라 토큰으로 정해야 한다"는 걸 배운 적이 있는데, 이번엔 그 교훈이
인가(권한 확인) 층에서 똑같이 반복됐다. 같은 원리가 층을 바꿔가며 나타난다는 걸 보고, 앞으로 "이건
무엇을 기준으로 대상을 정하지?"를 습관처럼 먼저 묻기로 했다.
