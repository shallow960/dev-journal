# 2026-07-20 서버에 값을 넣었는데 왜 안 먹지

> 태그: #트러블슈팅 #배포 #설정 #진단

## 무엇을 / 왜
자동 배포로 새 화면을 올렸더니 서비스 대문 페이지가 열리지 않고 오류가 났다. 이상한 건 서버가 전부
정상이었다는 점이다. 컨테이너도 다 떠 있고, 상태 확인용 주소도 잘 응답했다. 그런데 사람이 접속하는
페이지만 안 열렸다.

## 어떻게 풀었나
원인은 "여기가 우리 서비스 대문 주소다"를 알려주는 설정 값이 비어 있던 것이었다. 그 값이 없으니
프로그램이 대문 주소를 사용자 쇼핑몰 주소로 착각했고, 그런 쇼핑몰은 없으니 화면을 못 그려 오류가 났다.

값을 넣으면 되겠거니 하고 서버 실행 설정에 값을 넣었는데 안 됐다. 여러 번 다시 해봐도 그대로였다.
찾아보고 나서야 알았다. **그 값은 실행할 때 읽는 게 아니라, 앱을 만들 때(빌드) 코드 안에 박혀버리는
종류**였다. 이미 만들어진 앱에는 나중에 값을 넣어줘도 소용이 없고, 앱을 만드는 시점에 넣어야 했다.

앱을 만들 때 값이 전달되도록 고쳤고, 혹시 값이 또 빠지더라도 최소한 대문인 줄은 알도록 기본값을
정해뒀다.

### 같은 실수를 한 번 더 했다
문제가 풀린 줄 알았는데, 그 뒤에 앱을 손으로 다시 만들 때 이번엔 **다른 값**을 빠뜨렸다. 브라우저의
요청을 서버로 넘겨줄 주소였다. 화면은 떴는데 로그인 같은 기능이 전부 죽었다. 원인이 같은 종류였다는 걸
알고 좀 부끄러웠다.

이때 원인을 찾은 방법이 도움이 됐다. 서버 기록을 아무리 봐도 로그인 관련 오류가 하나도 없었다.
그래서 서버에 직접 요청을 넣어봤더니 정상적으로 성공했다. 서버는 멀쩡한데 브라우저로 하면 실패한다는
뜻이니 중간 단계가 문제였고, 브라우저 요청을 서버로 넘겨주는 주소가 아무것도 없는 곳으로 박혀 있는 걸
찾아냈다. 앞의 문제와 뿌리가 같았다.

## 💡 배운 것
- 설정 값에는 **실행할 때 읽는 것**과 **앱을 만들 때 코드에 박히는 것**이 있다. 후자는 나중에 서버에
  넣어줘도 반영되지 않는다. 값이 안 먹을 때는 "이 값이 언제 정해지는가"부터 확인하면 빨리 좁혀진다.
- 여러 단계를 거치는 요청(브라우저 -> 중계 -> 서버)은 단계를 하나씩 끊어서 어디까지 잘 가는지 볼 수
  있다. 막연히 전체를 들여다보는 것보다 훨씬 빨랐다.
- 기록에 오류가 **없다는 것 자체**도 단서가 된다. 서버에 오류가 안 남았다는 건 요청이 거기까지 아예
  도달하지 않았다는 뜻이었다.
- 손으로 하는 작업은 조건을 빠뜨리기 쉽다. 자동화가 쓰는 값과 똑같이 맞춰놓고 하지 않으면, 자동으로
  할 때는 되던 게 손으로 할 때만 깨진다.

## 코드로 보기 (개념 재현 테스트)
"만들 때 박히는 값"과 "실행할 때 읽는 값"이 어떻게 다른지 더미로 재현한다.

```java
// 목적: 빌드 시점에 코드로 치환되는 값은, 실행 시점에 환경변수를 바꿔도 반영되지 않는다.
class BuildTimeVsRuntimeConfigTest {

    // 빌드할 때 값이 그대로 박혀버린 경우 (이미 문자열로 굳었다)
    static final String BAKED_AT_BUILD = "";      // 빌드 때 값이 비어 있었다면 영원히 빈 값

    // 실행할 때 읽는 경우 (그때그때 환경을 본다)
    static String readAtRuntime(Map<String, String> env) {
        return env.getOrDefault("PLATFORM_HOST", "");
    }

    @Test
    void 실행시점에_값을_넣어도_빌드때_박힌_값은_안_바뀐다() {
        // 배포 후 서버에 값을 제대로 넣어줬다고 가정
        Map<String, String> env = Map.of("PLATFORM_HOST", "example.com");

        // 실행 시점에 읽는 값은 반영된다
        assertThat(readAtRuntime(env)).isEqualTo("example.com");

        // 하지만 빌드 때 박힌 값은 그대로 비어 있다 - 여기서 오작동이 시작됐다
        assertThat(BAKED_AT_BUILD).isEmpty();
    }

    // 값이 비었을 때 곧바로 오작동하지 않도록, 기본값을 두는 편이 안전하다
    static String platformHost(String configured) {
        return (configured == null || configured.isBlank()) ? "example.com" : configured;
    }

    @Test
    void 값이_비어도_기본값으로_버틴다() {
        assertThat(platformHost(BAKED_AT_BUILD)).isEqualTo("example.com");
    }
}
```

## 소감
오류 화면만 보면 "서버가 죽었나" 싶었는데, 실제로는 프로그램이 주소를 잘못 알아본 문제였다. 겉으로
보이는 증상과 진짜 원인이 이렇게 멀 수 있다는 걸 겪었다.

같은 종류의 실수를 짧은 간격으로 두 번 한 게 특히 기억에 남는다. 처음엔 운이 없었다고 생각했는데,
돌아보니 손으로 만들 때 조건을 맞춰 놓지 않은 게 원인이었다. 앞으로 설정 값을 다룰 때는 이게 언제
정해지는 값인지부터 확인하고, 손으로 하는 작업도 자동화와 같은 조건으로 맞춰놓고 시작해야겠다.
