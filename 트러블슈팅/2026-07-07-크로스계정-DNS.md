# 2026-07-07 도메인 DNS가 클라우드 콘솔에서 안 보이던 문제

> 태그: #트러블슈팅 #클라우드 #DNS #인프라

## 무엇을
도메인을 사고 DNS 설정(호스팅 영역)도 만들었는데, 정작 우리 프로젝트 클라우드 콘솔에서 그 호스팅 영역이 아무리 찾아도 안 보였다.

## 어떻게 풀었나
처음엔 지역(리전) 설정 문제인가 싶어 이리저리 바꿔봤지만 안 보였다. 알고 보니 DNS는 지역과 무관한 전역 서비스라, 리전은 애초에 원인이 될 수 없었다. 그래서 범위를 "계정"으로 좁혔다.

원인은 계정 격리였다. 클라우드 계정이 조직 구조로 나뉘어 있었고 - 전체를 관리하는 최상위 계정과, 프로젝트별 리소스가 담기는 워크로드 계정이 따로 있었다. DNS 호스팅 영역은 관리 계정에 만들어져 있었는데(관리자로 로그인하는 습관 탓), 서버와 DB 같은 실제 리소스는 워크로드 계정에 있었다. 클라우드는 계정끼리 완전히 격리돼서, 한 계정에서 만든 리소스는 다른 계정 콘솔엔 아예 안 보이는 게 정상이다.

해결은 이렇게 했다. DNS 호스팅 영역은 계정 간 "이동" 기능이 없어서, 리소스가 있는 워크로드 계정에 호스팅 영역을 새로 만들고 도메인 등록업체의 네임서버를 그쪽으로 위임했다. 이때 순서가 중요했다 - 옛 설정을 먼저 지우면 도메인이 잠깐 먹통이 되므로, 새 쪽을 완성하고 네임서버를 넘긴 뒤 전파를 확인하고 나서 옛 것을 정리했다. 그리고 서버에는 재시작해도 안 바뀌는 고정 IP를 먼저 붙여, DNS가 가리키는 대상이 흔들리지 않게 했다.

## 💡 배운 것
- "분명히 만들었는데 콘솔에 안 보인다"면 지역보다 먼저 계정을 의심하자. DNS 호스팅 영역 같은 자원은 계정마다 격리되고, 특히 조직 구조에선 관리 계정과 워크로드 계정을 헷갈려 엉뚱한 계정에 리소스를 만들기 쉽다.
- DNS 호스팅 영역은 계정 간 이동이 안 된다. 옮기려면 새로 만들고 네임서버를 위임하는 수밖에 없고, 이때 "새 것 완성 -> 전환 -> 전파 확인 -> 옛 것 삭제" 순서를 지켜야 다운타임이 없다.
- DNS를 붙이기 전에 서버 주소부터 고정(고정 IP 할당)해야 한다. 안 그러면 서버를 재시작할 때 주소가 바뀌어 DNS가 깨진다.

## 코드로 보기 (검증 명령)
네임서버 위임과 주소 전파가 실제로 됐는지, 로컬 캐시에 속지 않게 공개 리졸버로 확인한다.

```bash
# 1) 네임서버가 새 계정 쪽으로 위임됐는지 (등록업체 -> 새 호스팅 영역)
#    공개 리졸버(8.8.8.8)로 조회해 내 PC 캐시 영향을 배제
dig +short NS example.com @8.8.8.8
# -> ns-xxx.example-dns.net ...  (새로 만든 호스팅 영역의 네임서버가 나와야 정상)

# 2) 도메인이 고정 IP를 가리키는지
dig +short A example.com @8.8.8.8
# -> 203.0.113.10   (서버에 붙인 고정 IP)

# 3) 전파 완료 + 서버 응답까지 한 번에 확인
curl -s -o /dev/null -w "%{http_code}\n" https://example.com
# -> 200
```

## 소감
리전 설정만 뒤지다가 한참 헤맸다. "안 보인다 = 없는 게 아니라 다른 방(계정)에 있다"로 관점을 바꾸니 바로 풀렸다. 클라우드 계정을 조직으로 왜 나누는지, 관리 계정과 워크로드 계정을 왜 구분하는지 이번에 몸으로 이해했다.
