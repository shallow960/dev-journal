# 2026-07-22 검색 결과가 텅 비던 건, 페이지 번호가 한 칸 밀려서였다

> 태그: #트러블슈팅 #페이지네이션 #경계값

## 무엇을 / 왜
검색 화면을 열었는데 상품이 하나도 안 나왔다. 이상한 건 **홈 화면은 상품이 잘 나온다**는 거였다. 같은
상품 목록을 쓰는데 검색만 텅 비었다.

## 어떻게 풀었나
잘 되는 홈과 안 되는 검색의 차이를 좁혀 가다 보니, 검색만 **페이지 번호를 서버에 넘기고** 있었다.

목록을 페이지 단위로 부를 때 - 뒷단(서버)은 첫 페이지를 **0번**으로 세는데, 화면은 사람이 보기 좋게
**1번**부터 센다. 화면이 "1페이지 주세요" 하고 1을 그대로 넘기면, 서버는 그걸 "맨 앞(0번)을 건너뛴
두 번째 페이지"로 읽는다.

마침 상품이 한 페이지 분량(8개 < 한 페이지 12개)뿐이라, 두 번째 페이지엔 아무것도 없어서 화면이 텅 빈
거였다. 홈은 페이지 번호를 안 넘기고 기본값(0번)을 써서 우연히 멀쩡했다.

고치는 건 간단했다. **화면 번호(1부터)를 서버에 넘기기 직전에 한 칸 빼서(0부터) 맞춘다.** 화면에
보이는 번호는 1부터 그대로 두고, 넘기는 값만 변환한다.

## 코드로 보기 (개념 재현)
```ts
// 화면(사람용)은 1페이지부터, 서버(API)는 0페이지부터 센다.
// 이 경계를 안 맞추면 첫 진입(1페이지)이 서버엔 "두 번째 페이지"가 되어 텅 빈다.

const uiPage = Number(params.page ?? 1);   // 사람이 보는 번호: 1, 2, 3 ...

const list = await fetchGoods({
  // ...검색어 / 정렬 등...
  page: uiPage - 1,                         // ★서버로 넘길 땐 0-base로 변환
  size: PAGE_SIZE,
});

// 페이지네이션 UI는 그대로 1-base로 그린다(사용자에겐 1페이지로 보임)
```

## 💡 배운 것
- **"어디서부터 세는가(0-base vs 1-base)"가 한 칸만 어긋나도 결과가 통째로 빈다.** 특히 데이터가 한
  페이지 분량이면 "두 번째 페이지"가 비어서, 증상이 "일부 누락"이 아니라 "아예 안 나옴"으로 크게 난다.
- **경계 변환은 한 곳에서만.** 화면은 1-base, 서버는 0-base로 각자 자연스럽게 두되, 넘기는 지점에서
  딱 한 번 변환하면 헷갈리지 않는다.
- **"비슷한데 하나만 이상하다"면 그 하나의 차이부터 좁힌다.** 홈은 되고 검색만 안 되던 차이가 곧 원인
  (페이지 번호를 넘기느냐)이었다.

## 소감
"안 나온다"고 하면 데이터가 없거나 권한 문제부터 의심하기 쉬운데, 정작 원인은 숫자를 세는 시작점 하나
였다. 잘 되는 화면(홈)과 나란히 놓고 차이를 비교한 게 지름길이었다.
