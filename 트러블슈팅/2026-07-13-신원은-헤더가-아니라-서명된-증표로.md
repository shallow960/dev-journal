# 2026-07-13 신원은 헤더가 아니라 서명된 증표로

> 태그: #트러블슈팅 #보안 #접근통제

## 무엇을 / 왜
주문 관련 화면을 살펴보다가, 로그인한 사용자가 **주문 번호만 알면 남의 주문을 통째로 볼 수 있다**는
걸 발견했다. 주문에는 이름, 주소, 연락처, 결제 정보가 다 들어 있으니 개인정보가 그대로 새는
셈이었다. 좀 놀랐다.

원인은 두 가지가 겹쳐 있었다.

- **누구인지를 클라이언트가 보내는 값으로 받고 있었다.** "나는 누구다"를 요청 헤더나 파라미터에 담아
  보내면 서버가 그걸 그대로 믿었다. 공격자가 그 값을 피해자 것으로 바꾸면 피해자인 척할 수 있는 거다.
- **이게 네 것 맞냐는 확인이 없었다.** 주문을 번호로만 찾아 주고, "이게 요청한 사람 것이 맞나"를
  안 봤다.

## 어떻게 풀었나
"이 사람이 누구냐"를 판단하는 근거를 클라이언트가 주는 값이 아니라 서버가 발급한 것으로 바꿨다.

- **신원은 서명된 로그인 증표에서만 읽었다.** 로그인할 때 서버가 서명해서 준 증표(토큰) 안의 사용자
  식별자만 신원으로 쓰고, 클라이언트가 보낸 값은 아예 무시했다. 증표는 서버가 서명한 거라 위조할 수
  없어서, 남의 식별자를 넣은 증표를 만들 수가 없다.
- **조회를 본인 것만 보이게 좁혔다.** 주문을 가져올 때 "요청한 사람 소유"라는 조건을 같이 걸고,
  남의 것이면 "권한 없음"이 아니라 **"없는 것"처럼(404)** 처리했다. 있는지 없는지조차 안 알려주는 거다.
- 이 규칙을 주문뿐 아니라 장바구니, 결제 미리보기처럼 개인 정보를 다루는 경로에도 똑같이 적용했다.

고친 뒤에 로그인 안 한 접근, 남의 식별자를 넣은 접근, 다른 사람 증표로 하는 접근이 전부 막히고
본인 것만 열리는 걸 직접 호출해 보며 확인했다.

## 💡 배운 것
- 클라이언트가 보내주는 값은 뭐든 그대로 믿으면 안 된다는 걸 이번에 확실히 알았다. 헤더든
  파라미터든 사용자가 마음대로 바꿀 수 있는 값으로 "너 누구야"를 정하면 안 되는 거였다. 신원은
  서버가 서명해서 발급한 것만 믿어야 했다.
- "로그인했다"랑 "이걸 볼 자격이 있다"가 다른 문제라는 게 처음엔 헷갈렸다. 로그인 여부만 확인하고
  넘어갔더니, 로그인한 사람이면 아무나 남의 주문을 볼 수 있었다. 이 둘은 따로따로 확인해야 하는
  거였다.
- 남의 자원에 "권한 없음"이라고 답하는 게 오히려 "그게 있긴 있구나"를 알려주는 셈이라는 게
  의외였다. 그래서 아예 "없음(404)"으로 답해서 존재하는지조차 못 알게 하는 게 더 안전하다고 한다.

## 코드로 보기 (개념 재현 테스트)
"클라이언트가 준 신원(취약)"과 "서명 증표 신원 + 소유권 검사(안전)"의 차이를 재현한다.

```ts
// 취약: 신원을 클라이언트가 보낸 값에서 읽는다. 소유권 검사도 없다.
function getOrderInsecure(req) {
  const userId = req.headers["x-user-id"];      // 클라이언트가 채워 보낸 값 = 조작 가능
  return db.findOrderById(req.params.id);         // 소유권 확인 없음 = 아무 주문이나 반환
}

// 안전: 신원은 서버가 서명한 증표에서만, 접근은 본인 소유로 스코프.
function getOrderSecure(req) {
  const userId = req.auth.subject;                // 서명된 토큰의 subject = 위조 불가
  const order = db.findOrderById(req.params.id);
  if (!order || order.ownerId !== userId) {
    throw new NotFound();                          // 남의 것은 "없는 것"으로 취급(존재 은폐)
  }
  return order;
}

// 왜 헤더가 위험한가: 공격자는 헤더 한 줄만 바꾸면 피해자 행세가 된다.
//   x-user-id: <피해자-식별자>   ->  취약 버전은 그대로 남의 주문을 내준다.
// 서명 토큰은 서버 키로 서명돼 있어, 남의 식별자를 담은 토큰을 만들 수 없다.
```

## 소감
"헤더로 사용자를 넘기자"는 초기 배선이 개발할 땐 편해서 그냥 썼는데, 그 편함이 그대로 뒷문이 될
줄은 몰랐다. 신원을 다룰 때는 "이 값을 사용자가 마음대로 바꾸면 어떻게 되지?"부터 떠올려야 한다는
걸, 직접 재현해 보면서 확실히 알게 됐다.
