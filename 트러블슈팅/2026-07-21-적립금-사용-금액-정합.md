# 2026-07-21 적립금은 쌓는 것보다 "쓰는 것"이 훨씬 얽혀 있었다

> 태그: #트러블슈팅 #결제 #환불 #데이터모델

## 무엇을 / 왜
손님 적립금(포인트)을 만들었다. 구매하면 일정 비율을 쌓아주고, 결제할 때 그 적립금을 쓸 수 있게 하는
기능이다. 쌓기(적립)는 단순했는데, **쓰기(차감)** 가 훨씬 복잡했다. 서로 다른 세 곳이 각자 다른 금액을
봐야 했기 때문이다.

- **카드 청구**: 적립금을 쓰면 실제로 카드로 빠지는 금액이 그만큼 줄어야 한다.
- **환불**: 나중에 취소·환불하면 카드는 "줄어든 금액"만 돌려주고, 쓴 적립금은 **따로** 되돌려줘야 한다.
- **정산(입점사에게 주는 돈)**: 적립금과 무관하게 **원래 상품값** 기준이어야 한다. 적립금은 플랫폼이
  부담한 로열티일 뿐, 물건을 판 입점사 몫을 깎는 게 아니다.

이 셋이 같은 주문을 놓고 서로 다른 금액을 봐야 하니, 한 군데라도 기준이 어긋나면 돈이 안 맞는다.

## 어떻게 풀았나
핵심은 **"실제로 청구할 금액"이라는 기준값을 하나로 모은 것**이다.

- `실 청구액 = 상품값 - 쓴 적립금` 을 주문이 스스로 계산해서 갖게 했다.
- **결제·금액 검증·취소·환불이 전부 이 하나의 값을 보도록** 통일했다. (외부 결제창에서 돌아온 금액이
  맞는지 확인할 때도, 취소·환불로 되돌릴 금액을 정할 때도 이 값을 쓴다.)
- **정산만은 상품값(품목 금액)을 그대로** 본다. 적립금은 여기에 손대지 않는다.

원자성도 중요했다.

- 적립금 차감은 **주문 생성과 한 트랜잭션으로 묶었다.** 주문이 중간에 실패하면 차감도 함께 없던 일이
  되고, 잔액이 부족하면 결제 전에 주문 자체가 실패한다 - 그래야 "돈은 덜 냈는데 포인트는 안 빠진"
  같은 어긋남이 안 생긴다.
- 취소·환불 때는 쓴 적립금을 **자동 복원**하되, 주문당 1회만 되도록(멱등) 이미 복원했는지 확인하고
  넘어가게 했다. 두 번 눌러도 두 번 안 돌려준다.
- 적립(쌓기)도 "실제로 낸 금액" 기준으로 바꿨다. 적립금으로 깎인 부분에까지 또 적립해주면 안 되니까.

실제로 100포인트를 쓴 주문을 넣어 청구액이 정확히 그만큼 줄어드는 것, 그 주문을 취소해서 카드
취소액과 포인트 복원이 각각 맞아떨어져 잔액이 원래대로 돌아오는 것까지 확인했다. 기록에는
`적립(+) → 사용(-) → 취소 복원(+)` 이 한 줄씩 남아 앞뒤가 맞았다.

## 💡 배운 것
- **돈이 얽힌 기능의 최대 위험은 "여러 곳이 같은 금액을 서로 다르게 계산"하는 것이다.** 계산을
  여기저기 흩지 말고 기준 금액을 **하나(single source of truth)** 로 모으면 어긋날 여지가 크게 준다.
- **차감·복원은 원래 거래와 원자적으로 묶어라.** 실패하면 함께 롤백되게, 중복 실행은 멱등하게. 안 그러면
  "한쪽만 반영된" 상태가 남는다.
- **사용자가 낸 돈(카드)과 플랫폼이 부담한 것(적립금)은 분리해서 각자 되돌려라.** 환불은 카드 취소 +
  포인트 복원의 두 갈래이고, 정산은 둘 다와 무관하게 상품값을 본다.
- 쌓아주는 기준도 "쓴 만큼 빼고 실제로 낸 금액"으로 - 안 그러면 안 낸 돈에까지 적립해주는 셈이 된다.

## 코드로 보기 (개념 재현)
"기준 금액 하나로 통일 + 차감을 주문과 원자적으로 + 복원은 멱등"을 개념만 재현한 골격이다. 실제 내부
이름·테이블은 뺐고 값은 더미다.

```java
// 주문이 스스로 "실 청구액"을 계산해서 갖는다. 결제/검증/취소/환불이 모두 이 값을 본다.
class Order {
    long goodsTotal;   // 상품값 합계
    long pointsUsed;   // 쓴 적립금
    long payable() { return goodsTotal - pointsUsed; }   // ★단일 기준값
}

// [체크아웃] 주문 생성과 적립금 차감을 한 트랜잭션으로 묶는다.
@Transactional
CheckoutResult checkout(Cart cart, long pointsToUse) {
    if (pointsToUse > cart.goodsTotal()) throw new BadRequest("적립금이 상품값 초과");

    Order order = Order.place(cart, pointsToUse);
    points.spend(cart.userId(), order.id(), pointsToUse); // 잔액 부족이면 여기서 실패 -> 주문째 롤백
    pay(order.payable());                                 // 카드 청구는 "실 청구액"으로
    return new CheckoutResult(order);
}

// [취소/환불] 카드는 실 청구액만 되돌리고, 쓴 적립금은 따로 복원(주문당 1회, 멱등).
@Transactional
void cancel(Order order) {
    refundCard(order.payable());                          // 카드: 낸 만큼만
    points.restoreForOrder(order.userId(), order.id());  // 포인트: 쓴 만큼 복원
    // 정산은 여기서 건드리지 않는다 - 상품값 기준으로 별도 처리
}

// 복원은 "이미 복원했으면 넘어가기"로 멱등하게. 두 번 눌러도 두 번 안 돌려준다.
void restoreForOrder(UUID userId, UUID orderId) {
    long spent = ledger.spentFor(orderId);
    if (spent <= 0 || ledger.alreadyRestored(orderId)) return;
    ledger.add(userId, orderId, +spent, "취소 복원");
}
```

## 소감
"적립금 쓰기"라고 하면 그냥 잔액에서 빼면 될 것 같았는데, 실제로는 카드·환불·정산·적립이 서로 다른
금액을 봐야 하는 문제였다. 각자 알아서 계산하게 두면 언젠가 어긋난다는 확신이 들어서, 기준값을
하나로 모으는 데 시간을 제일 많이 썼다. 그러고 나니 "이건 어떤 금액을 봐야 하지?"로 헷갈릴 일이
사라졌다. 돈을 다루는 코드에선 편의보다 정합이 먼저라는 걸 체감했다.
