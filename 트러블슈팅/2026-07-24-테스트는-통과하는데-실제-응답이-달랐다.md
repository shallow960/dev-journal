# 2026-07-24 테스트는 통과하는데 실제 서비스에선 다른 응답이 왔다

> 태그: #트러블슈팅 #서버 #테스트 #배포검증

## 무엇을 / 왜

로그인 없이 누구나 호출할 수 있는 창구(오류 수집, 문의 신청 같은)에 아무 제한이 없었다. 한 사람이
원하는 만큼 요청을 쏟아부으면 저장소가 계속 불어나는 상태라, "너무 자주 부르면 막고, 너무 큰 내용은
거절한다"는 방어를 요청 앞단에 넣었다.

테스트는 다 통과했고 배포도 정상이었다. 그런데 배포 후에 실제 서비스 주소로 큰 요청을 직접 보내
봤더니, 예상한 "내용이 너무 큽니다"(413)가 아니라 **"로그인이 필요합니다"(401)** 가 돌아왔다.

## 어떻게 풀았나

먼저 범위를 좁혔다. 같은 주소에 작은 요청을 보내면 평소 응답이 정상으로 오는데, 큰 요청만 다른 답이
왔다. 요청 크기 말고는 달라진 게 없으니 방금 넣은 방어가 원인이라는 게 분명해졌다.

원인은 **거절하는 방식**에 있었다. 요청을 막을 때 흔히 쓰는 것이 "이 상태코드로 오류를 보내라"고
컨테이너에 위임하는 방식인데, 이게 내부적으로 **오류 전용 경로로 요청을 한 번 더 태운다.** 그 재진입이
인증 검사를 다시 거치면서, 내가 정한 상태코드가 **인증 실패로 덮여버린** 것이었다.

막히는 것 자체는 제대로 되고 있었다. 요청은 통과하지 못했고 데이터도 쌓이지 않았다. 다만 부른 쪽은
"왜 막혔는지"를 알 수 없었다. 그래서 위임 대신 **상태코드와 본문을 직접 확정해서 돌려주도록** 바꿨다.
같은 방식으로 쓰여 있던 다른 차단 코드도 함께 정리했다.

더 신경 쓰인 건 **왜 테스트가 못 잡았나** 였다. 테스트 도구는 그 "오류 전용 경로"를 실제로 태우지
않는다. 그래서 내가 정한 상태코드가 그대로 보였고, 실제 서버에서만 덮였다. 상태코드만 확인하는 검증은
이 차이를 영원히 못 잡는다는 뜻이다.

그래서 검증을 **응답 본문에 우리가 정한 사유가 들어 있는지** 확인하는 방식으로 바꿨다. 위임 방식은
본문을 우리가 쓰지 않으므로, 예전 방식으로 되돌리면 반드시 실패한다.

고쳐서 다시 배포한 뒤, 처음에 401이 오던 그 요청을 그대로 다시 보내 의도한 응답이 오는 것을 실제
서비스에서 확인했다. 정상 요청이 전과 똑같이 동작하는 것도 함께 봤다.

## 💡 배운 것

- **테스트 통과와 실제 동작은 다를 수 있다.** 테스트 도구는 실행 환경을 흉내 낼 뿐 전부 재현하지
  않는다. 특히 프레임워크가 내부에서 요청을 다시 태우는 구간은 흉내에서 빠지기 쉽다.
- **거절도 "어떻게" 거절하느냐가 동작을 바꾼다.** 처리를 위임하면 그 다음에 무슨 일이 일어나는지는
  내 손을 떠난다. 응답을 확정해서 돌려주면 중간에 바뀔 여지가 없다.
- **테스트가 못 잡은 결함을 만나면 고치는 데서 끝내면 안 된다.** "다음에도 못 잡을 것"이 그대로
  남는다. 무엇을 확인해야 그 실수가 잡히는지까지 바꿔야 한 번 겪은 값을 한다.
- **배포는 끝이 아니라 검증의 시작이다.** 실제 주소로 한 번 찔러보지 않았다면 이건 조용히 남았을 것이다.

## 코드로 보기 (개념 재현 테스트)

거절을 위임하면 본문이 비어 원인이 남지 않고, 직접 쓰면 사유가 보존되는 것을 개념만 재현했다. 실제
내부 이름은 뺐고 값은 더미다.

```java
// 목적: 요청을 막을 때 "오류 처리를 위임"하면 본문이 비고, 컨테이너가 오류 전용 경로로 요청을
//       다시 태우면서 상태코드가 뒤바뀔 여지가 생긴다.
//       상태코드와 본문을 직접 확정하면 그 여지가 없고, 막힌 사유도 호출자에게 남는다.
class RejectResponseTest {

    // (취약) 컨테이너에 "이 코드로 오류를 내줘"라고 넘긴다.
    //        -> 오류 전용 경로 재진입 -> 그 경로가 인증 검사를 다시 거치면 상태코드가 덮인다.
    static void rejectByDelegating(HttpServletResponse response) throws IOException {
        response.sendError(429, "Too many requests");
    }

    // (수정) 상태코드와 본문을 여기서 확정한다. 재진입이 없으니 중간에 바뀌지 않는다.
    static void rejectDirectly(HttpServletResponse response) throws IOException {
        response.setStatus(429);
        response.setContentType("application/json;charset=UTF-8");
        response.getWriter().write("{\"code\":\"TOO_MANY_REQUESTS\"}");
        response.flushBuffer();
    }

    @Test
    void 위임하면_본문이_비어_막힌_사유가_남지_않는다() throws Exception {
        MockHttpServletResponse response = new MockHttpServletResponse();
        rejectByDelegating(response);

        assertThat(response.getStatus()).isEqualTo(429);
        // 상태코드만 보면 통과한다 - 그래서 이 검증만으로는 실제 서버의 덮어쓰기를 못 잡았다.
        assertThat(response.getContentAsString()).isEmpty();
    }

    @Test
    void 직접_쓰면_사유가_본문에_남는다() throws Exception {
        MockHttpServletResponse response = new MockHttpServletResponse();
        rejectDirectly(response);

        assertThat(response.getStatus()).isEqualTo(429);
        // 본문까지 확인한다 - 위임 방식으로 되돌리면 이 줄에서 반드시 실패한다.
        assertThat(response.getContentAsString()).contains("TOO_MANY_REQUESTS");
    }
}
```

## 소감

"테스트 초록이면 됐지"라고 넘겼으면 이건 안 보였을 것이다. 배포하고 실제 주소로 한 번 찔러본 게
전부인데 거기서 나왔다. 그리고 원인을 찾은 뒤에 더 중요한 질문이 남았다 - "왜 테스트는 통과했지?"
그 질문에 답하지 않으면 다음에도 같은 종류를 놓친다는 걸 알게 돼서, 고치는 것보다 검증 방식을
바꾸는 데 시간을 더 썼다. 앞으로 방어 코드를 넣을 땐 "이게 실제로 그 응답을 내는가"를 배포 후에
직접 확인하는 걸 절차로 넣기로 했다.
