# 2026-07-24 아침에는 깨지고 점심에는 통과하는 테스트

> 태그: #트러블슈팅 #시간대 #테스트 #날짜계산

## 무엇을 / 왜

아침에 전체 테스트를 돌렸는데 만료 처리 관련 테스트 하나가 실패했다. 그날 내가 고친 건 전혀 다른
부분이라 이상해서, 내 수정을 잠시 빼고 원래 코드로 돌려봤더니 **거기서도 똑같이 실패**했다. 내가
만든 게 아니라 원래 있던 문제였다.

그런데 몇 시간 뒤, **아무것도 안 고쳤는데 같은 테스트가 통과**했다.

## 어떻게 풀었나

"시간이 지나니 통과한다"에서 시간대 문제라는 감이 왔다. 확인해 보니 "오늘이 며칠인가"를 정하는
기준이 **세 군데에서 서로 달랐다.**

- 만료를 판정하는 코드: 세계 표준시(UTC)
- 데이터베이스에 직접 날짜를 묻는 부분: 접속 설정을 따라감
- 주문번호에 찍히는 날짜: 서버 기본값을 따라감

서비스가 영업하는 지역이 표준시보다 앞서 있으면(예: +9시간), **지역 시각 자정부터 오전 9시까지는
표준시로 아직 어제다.** 그래서 그 사이에는 이런 일이 벌어지고 있었다.

- 만료일이 지난 쿠폰이 **최대 9시간 더 사용**되고, 만료 처리 배치도 그만큼 늦게 돌았다.
- 주문번호가 **어제 날짜로 발번**됐다. 손님에게 그대로 보이는 값인데.

여기서 한 가지를 구분하고 나서야 정리가 됐다. **"시각"과 "날짜"는 다른 문제다.**

- 로그인 유효기간, 인증번호 유효창처럼 **순간**을 다루는 값은 표준시가 맞다. 어디서 보든 같은 순간을
  가리켜야 하니까. 이건 그대로 뒀다.
- "만료일", "사용 시작일"처럼 **손님이 달력을 보고 이해하는 값**은 서비스가 영업하는 지역의 시간대여야
  한다. 이것만 옮겼다.

그 구분이 코드에 드러나도록 "영업 달력"이라는 개념을 따로 만들고, 날짜를 쓰는 곳이 전부 그걸 거치게
했다. 데이터베이스에 직접 날짜를 묻던 조회도 계산한 날짜를 넘기는 방식으로 바꿨다. 자정에 돌 의도로
적어둔 배치들도 실제로는 지역 시간 낮에 돌고 있어서, 실행 시간대를 영업 기준으로 맞췄다.

## 💡 배운 것

- **"가끔 깨지는 테스트"는 넘기면 안 된다.** 이번엔 테스트가 유난스러운 게 아니라 실제로 사용자에게
  영향이 가는 결함을 가리키고 있었다. 재현이 어렵다고 넘겼으면 그대로 남았을 것이다.
- **시각과 달력 날짜는 다르게 다뤄야 한다.** 둘을 같은 기준으로 묶으면 한쪽이 반드시 어긋난다.
  순간을 가리키는 값은 표준시, 사람이 달력으로 읽는 값은 지역 시간대.
- **"오늘"을 정하는 기준이 여러 군데면 언젠가 갈라진다.** 코드·데이터베이스·서버 기본값이 각자 답하고
  있었는데, 평소엔 우연히 같아서 아무도 몰랐다. 기준은 한 곳에서 답하게 만들어야 한다.
- **시각에 따라 결과가 달라지는 테스트는 검증이 아니다.** 통과해도 통과한 건지 우연인지 알 수 없다.

## 코드로 보기 (개념 재현 테스트)

같은 순간인데 기준 시간대에 따라 날짜가 하루 다르고, 그 차이가 만료 판정을 뒤집는 것을 고정된 시각으로
재현했다. 실제 내부 이름은 뺐고 값은 더미다.

```java
// 목적: "오늘"을 표준시로 잡으면, 표준시보다 앞선 지역에서는 자정~오전 사이에 날짜가 하루 뒤처진다.
//       그 구간에서 만료 판정이 뒤집혀 지난 쿠폰이 아직 살아있게 된다.
//       시각을 고정했기 때문에 언제 돌려도 같은 결과가 나온다(= 검증이 된다).
class BusinessDateTest {

    // 표준시 2026-07-23 23:30 == 지역시(+9) 2026-07-24 08:30 - 같은 순간, 다른 날짜.
    static final Clock FIXED = Clock.fixed(Instant.parse("2026-07-23T23:30:00Z"), ZoneOffset.UTC);
    static final ZoneId BUSINESS_ZONE = ZoneId.of("Asia/Seoul"); // 영업 지역(예시)

    @Test
    void 같은_순간인데_기준에_따라_날짜가_하루_다르다() {
        assertThat(LocalDate.now(FIXED)).isEqualTo(LocalDate.parse("2026-07-23"));                    // 표준시
        assertThat(LocalDate.now(FIXED.withZone(BUSINESS_ZONE))).isEqualTo(LocalDate.parse("2026-07-24")); // 영업 달력
    }

    @Test
    void 표준시로_판정하면_만료일이_지난_쿠폰이_아직_살아있다() {
        LocalDate 만료일 = LocalDate.parse("2026-07-23"); // 어제(영업 달력 기준)까지가 사용 기한

        // (취약) 표준시 기준: "오늘"이 아직 7/23 이라 만료로 안 본다 -> 최대 9시간 더 쓰인다.
        assertThat(만료일.isBefore(LocalDate.now(FIXED))).isFalse();

        // (수정) 영업 달력 기준: "오늘"이 7/24 이므로 정상적으로 만료된다.
        assertThat(만료일.isBefore(LocalDate.now(FIXED.withZone(BUSINESS_ZONE)))).isTrue();
    }
}
```

## 소감

처음엔 "왜 지금 깨지지?" 하고 넘어갈 뻔했다. 몇 시간 뒤에 저절로 통과하는 걸 보고서야 이게 코드가
아니라 시간의 문제라는 데 생각이 닿았다. 재현이 안 되는 게 아니라 **재현되는 시간대가 따로 있었던**
것이다.

고치면서 제일 도움이 된 건 "시각이냐 날짜냐"를 나눈 것이었다. 그 전에는 전부 시간대 문제로 뭉뚱그려
봐서 어디까지 바꿔야 할지 감이 안 왔는데, 나누고 나니 건드릴 곳과 두어야 할 곳이 분명해졌다. 앞으로
날짜를 쓸 때마다 "이건 순간인가, 사람이 달력으로 읽는 값인가"를 먼저 묻기로 했다.
