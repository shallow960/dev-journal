# 2026-07-02 결제 "성공코드", 문서가 맞다던 값이 실제와 달랐다

> 태그: #트러블슈팅 #결제 #검증

## 무엇을 / 왜
구독 자동결제를 실제 카드로 청구하는 단계까지 붙였다. 앞서 결제창·카드 인증·자동결제 키(빌링키) 발급은 끝냈고, 이번엔 그 키로 **실제 카드에 매달 돈을 청구**하는 승인 단계 차례였다. 각 결제 API의 "성공 판정 코드"는 처음엔 문서와 샘플을 보고 가정해 뒀다.

## 어떻게 풀었나
가정에 의존하지 않고 테스트 환경에서 실제 카드로 청구를 한 건씩 태워봤다. 그랬더니 두 가지가 걸렸다.

**1. 할부 개월 필드 규격 (자릿수 하나 차이)**
일시불을 한 자리 `"0"`으로 보냈더니 결제사가 "잘못된 할부개월수"로 거부했다. 규격은 두 자리 `"00"`이었다. 문서만 읽었을 땐 안 보이던, 태워봐야 나오는 종류의 함정.

**2. 성공코드 불일치 (진짜 문제)**
실제 카드가 정상 청구돼서 승인번호까지 떨어졌는데, 내 코드가 "성공"이라고 가정한 코드값과 결제사가 실제로 돌려준 성공코드가 **서로 달랐다.** 그대로 뒀으면 **진짜 성공한 결제를 실패로 처리**할 뻔했다. 돈은 빠져나갔는데 시스템은 "실패"로 기록하는, 결제에서 제일 위험한 종류의 미스매치다.

실제 응답값을 기준으로 성공코드를 교정하고, 발급 → 승인 → 취소·해지까지 전 구간을 실거래로 다시 한 바퀴 돌려 확정했다.

## 💡 배운 것
- **실패 처리만큼 "성공 판정"도 검증 대상이다.** 다들 실패 케이스는 신경 쓰지만, 성공코드를 잘못 가정하면 정상 결제를 실패로 오판한다. 사용자에겐 "결제 실패"로 보이는데 실제 돈은 나간, 최악의 조합.
- **외부 연동의 성공/실패 경계값은 문서가 아니라 실제 응답으로 확정한다.** "문서에 이렇게 써 있다"와 "실제로 이 값이 왔다"는 다르다. 특히 돈이 오가는 곳에선 가정 금지.
- **자릿수·포맷 같은 사소한 규격 하나가 전체를 막는다.** `"0"`과 `"00"`, 한 글자 차이. 이런 건 읽어서가 아니라 태워봐야 드러난다.

## 코드로 보기 (개념 재현 테스트)
"성공 여부를 실제 응답 코드 기준으로 판정한다"와 "할부 필드를 규격 포맷으로 맞춘다"는 두 지점을 자기완결 테스트로 재현한다.

```java
// 목적: 결제 승인 응답을 "성공/실패"로 판정하는 로직 검증.
// 핵심 함정 = 성공코드를 문서 가정값으로 잡으면 진짜 성공을 실패로 오판한다.
class ApprovalResultTest {

    // 결제사가 "청구 성공" 시 실제로 주는 코드 (값은 예시 더미)
    private static final String SUCCESS_CODE = "3001";

    // 승인 응답의 결과코드로 성공 여부를 판정
    private boolean isApproved(String resultCode) {
        return SUCCESS_CODE.equals(resultCode);
    }

    // 일시불/할부 개월수를 규격 포맷(두 자리)으로 정규화
    // "0"(한 자리)로 보내면 잘못된 할부개월수로 거부됨 -> "00"이 규격
    private String normalizeInstallment(String months) {
        return String.format("%02d", Integer.parseInt(months));
    }

    @Test
    void 실제_성공코드를_성공으로_인식한다() {
        // 라이브에서 실제 카드 청구가 성공했을 때 돌아온 코드
        assertThat(isApproved("3001")).isTrue();
    }

    @Test
    void 가정했던_성공코드는_실제로는_성공이_아니었다() {
        // 처음에 문서만 보고 성공이라 가정했던 코드값.
        // 이 값을 기준으로 뒀으면 '성공한 결제'가 실패로 처리됐다.
        assertThat(isApproved("F200")).isFalse();
    }

    @Test
    void 일시불_할부개월은_두자리_규격이라야_통과() {
        // 한 자리로 보내 거부당했던 값을 규격 포맷으로 교정
        assertThat(normalizeInstallment("0")).isEqualTo("00");
    }
}
```

## 소감
"성공했으니 성공으로 뜨겠지"를 의심하게 된 날이다. 실패 케이스만 방어하다가 성공 판정에서 발목 잡힐 뻔했다. 결제는 특히 "실제로 태워보기 전엔 아무것도 확정하지 않는다"를 몸에 새겼다. 이제 발급·승인·취소·환불 전 구간이 실거래로 검증됐으니, 다음은 이걸 자동청구 스케줄러에 얹어 매달 도는 흐름까지 확인할 차례다.
