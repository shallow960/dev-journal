# 2026-07-15 목록을 가볍게 하려다 편집 저장에서 데이터가 지워질 뻔

> 태그: #트러블슈팅 #데이터무결성

## 무엇을 / 왜
상품을 여러 개 묶어서 보여주는 기능(기획전)을 새로 만들었다. 목록 화면에서는 각 항목의 요약만 보여주면
되니까, 목록 응답에는 무거운 정보(어떤 상품들이 연결됐는지 전체 목록)를 빼고 가볍게 내려줬다. 여기까진
좋았다.

## 어떻게 풀었나 (문제 → 원인 → 해결)
편집 화면을 만들면서 문제가 보였다.
- **문제:** 편집 폼은 목록에서 받은 데이터로 채우는데, 목록엔 "연결된 상품 목록"이 빠져 있다. 그래서 편집
  폼을 열면 연결 상품이 비어 있는 상태가 된다. 여기서 제목만 살짝 고치고 저장하면, 서버에는 "연결 상품 =
  비어 있음"이 그대로 전송돼 원래 걸어둔 상품이 전부 지워진다.
- **원인:** 목록을 가볍게 하려고 뺀 정보가, 편집 폼의 초기값으로도 그대로 쓰이면서 "없음"으로 오인됐다.
  화면엔 아무 에러도 안 뜨고 저장은 성공하니, 눈치채기 전까지 조용히 데이터가 사라진다.
- **해결:** 편집 화면을 열 때는 목록 데이터에 기대지 않고, 그 항목의 상세를 따로 한 번 더 불러와 연결 상품
  까지 채운 다음 폼을 연다. 그리고 상세를 다 불러오기 전에는 저장 버튼을 막아, 빈 값으로 덮어쓰는 걸
  원천 차단했다.

## 💡 배운 것
- "목록은 가볍게"는 좋은 기본이지만, 그 가벼운 데이터를 편집의 초기값으로 재사용하면 "생략"이 "삭제"로
  둔갑할 수 있다.
- 눈에 보이는 에러가 없는 데이터 손실이 제일 위험하다. 저장이 성공했다는 것과 의도대로 저장됐다는 건
  다르다.
- 저장하는 화면에선 "이 값이 비어 있으면 그건 '지워라'인가, '안 바뀜'인가?"를 항상 구분해야 한다.

## 코드로 보기 (개념 재현 테스트)
요약(연결 목록 없음)으로 폼을 채워 저장하면 연결이 지워지고, 상세로 채우면 지켜지는 걸 더미로 재현한다.

```java
// 목적: 요약으로 편집·저장하면 기존 연결이 사라지고, 상세로 채우면 유지되는 함정을 보인다.
class EditPrefillTest {

    record Item(String title, List<String> linkedIds) {}

    // 서버 저장: 받은 값으로 통째로 덮어쓴다 (비어 있으면 비운다)
    static Item save(String title, List<String> linkedIds) {
        return new Item(title, linkedIds);
    }

    @Test
    void 요약으로_채워_저장하면_연결이_지워진다() {
        List<String> fromSummary = List.of();          // 목록 요약엔 연결 목록이 없음
        Item saved = save("제목만 수정", fromSummary);
        assertThat(saved.linkedIds()).isEmpty();        // 원래 걸어둔 상품이 사라짐 - 사고
    }

    @Test
    void 상세로_채워_저장하면_연결이_유지된다() {
        List<String> fromDetail = List.of("g1", "g2"); // 편집 열 때 상세를 따로 불러와 채움
        Item saved = save("제목만 수정", fromDetail);
        assertThat(saved.linkedIds()).containsExactly("g1", "g2"); // 유지됨
    }
}
```

## 소감
새 기능을 만드는 일보다, "목록은 가볍게"라는 당연해 보이는 최적화가 다른 화면(편집)과 만나는 지점에
사고가 숨어 있었다. 코드가 에러 없이 잘 돌아가고 저장도 성공해서, 테스트로 직접 "저장하면 이 값이
어떻게 되지?"를 확인해보지 않았으면 못 봤을 문제다. 앞으로 저장 기능을 만들 때는 빈 값이 "삭제"인지
"안 바뀜"인지부터 자문하는 습관을 들이기로 했다.
