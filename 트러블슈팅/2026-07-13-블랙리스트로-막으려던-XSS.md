# 2026-07-13 블랙리스트로 막으려던 XSS

> 태그: #트러블슈팅 #보안 #XSS

## 무엇을 / 왜
판매자가 쓴 상품 설명을 손님 화면에 그대로 보여 주는 곳이 있었다. 서식이 들어가는 글이라 HTML을
그리는데, 위험한 태그를 정규식으로 지우는 방식으로 막고 있었다 - `<script>`를 지우고, `on...=` 같은
이벤트 핸들러를 지우고, `javascript:` 링크를 지우는 식이었다.

문제는 이 방식이 **너무 쉽게 우회된다**는 거였다. 정규식이 "따옴표로 감싼 이벤트 핸들러"만 지웠는데,
공격자가 따옴표 없이 쓰면 그냥 통과했다.

```
<img src=x onerror=탈취코드>   ->  따옴표가 없어 정규식이 못 잡음 -> 통과
```

통과한 스크립트는 설명에 저장되고, **그 상품을 보는 손님들 브라우저에서 전부 실행**된다. 이렇게
한 번 저장돼서 두고두고 실행되는 걸 저장형 XSS(stored XSS)라고 한다. 판매자는 우리 직원이 아니라
완전히 믿을 수만은 없는 사람인데, 그 사람이 쓴 내용이 손님 브라우저에서 우리 사이트 권한으로 돌면
손님의 로그인 세션까지 노릴 수 있다.

## 어떻게 풀었나
막는 방향 자체를 뒤집었다.

- **"막을 걸 나열"하던 걸 "허용할 것만 남기는" 쪽으로 바꿨다.** 위험한 걸 하나하나 지우는 대신,
  안전한 태그랑 속성만 적어 둔 **허용 목록**을 통과시키고 나머지는 전부 지우는 방식으로 갔다. 목록에
  없는 `onerror`, `<script>`, `javascript:` 는 어떤 형태로 오든 그냥 사라진다.
- **정화하는 시점을 저장 전으로 옮겼다.** 화면에 그릴 때가 아니라 **저장하기 전에** 한 번 걸러서,
  저장된 데이터 자체가 안전하게 만들었다. 혹시 몰라 화면 쪽에도 한 겹 더 걸러 두었다.
- 우회하려는 입력(따옴표 없는 핸들러, svg, 인라인 데이터 링크 같은 것)이 실제로 무력화되고, 멀쩡한
  서식은 그대로 남는지 테스트로 확인해 두었다.

## 💡 배운 것
- 처음엔 위험한 태그를 하나씩 지우면 되는 줄 알았다. 그런데 딱 하나 빠뜨린 형태(따옴표 없는
  핸들러)로 그냥 뚫렸다. 막을 걸 전부 나열하는 건 애초에 불가능하고, 반대로 "안전한 것만 남기는"
  방식이 훨씬 든든하다는 걸 이번에 알았다. 목록에 없으면 모르는 공격도 기본적으로 걸러진다.
- 정화를 화면에 그릴 때가 아니라 저장하기 전에 해야 한다는 것도 이번에 배웠다. 화면에서만 막으면
  다른 화면이 그 데이터를 날것으로 받을 수 있어서, 데이터가 들어오는 길목에서 미리 막는 게 마음
  편했다(그래도 화면 쪽에 한 겹 더 두는 게 정석이라고 한다).
- "우리가 만든 화면이니까 안전하겠지"가 착각이었다. 판매자가 쓴 글이 손님 화면에서 실행되면, 결국
  남이 우리 사이트에 코드를 심은 거랑 똑같은 일이 벌어진다는 걸 알았다.

## 코드로 보기 (개념 재현 테스트)
블랙리스트가 우회되는 것과, 허용 목록이 우회 페이로드를 중화하는 것을 재현한다.

```ts
// 블랙리스트: "위험한 것"을 지운다. 따옴표 있는 핸들러만 잡아 우회당한다.
function sanitizeBlacklist(html: string) {
  return html
    .replace(/<script[\s\S]*?<\/script>/gi, "")
    .replace(/\son\w+="[^"]*"/gi, "");   // 따옴표로 감싼 핸들러만 제거
}

// 허용 목록: "안전한 것"만 남긴다. 목록에 없는 건 형태 불문 제거.
const ALLOWED_TAGS = ["p", "b", "i", "ul", "li", "a", "img"];
const ALLOWED_ATTRS = { a: ["href"], img: ["src", "alt"] }; // href/src 는 http(s)만
function sanitizeAllowlist(html: string) {
  // (실제로는 검증된 HTML 정화 라이브러리를 쓴다 - 여기선 개념만)
  return allowlistClean(html, ALLOWED_TAGS, ALLOWED_ATTRS);
}

// 우회 페이로드: 따옴표 없는 이벤트 핸들러
const attack = "<img src=x onerror=steal(document.cookie)>";

// 블랙리스트는 통과시키고(취약), 허용 목록은 onerror 를 제거한다(안전).
expect(sanitizeBlacklist(attack)).toContain("onerror");    // 뚫림
expect(sanitizeAllowlist(attack)).not.toContain("onerror"); // 막힘
```

## 소감
"위험한 걸 지우면 되지"라는 생각이 직관적이라 나도 모르게 먼저 손이 갔다. 근데 보안에서는 그 직관이
자주 함정인 것 같다. 하나만 빠뜨려도 전부 무너지는 곳에서는, 막을 걸 나열하기보다 허용한 것만
남기는 쪽이 왜 안전한지, 실제로 뚫리는 걸 재현해 보고 나서야 와닿았다.
