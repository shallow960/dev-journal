# 2026-07-01 로그인은 되는데 브라우저에서만 세션이 안 잡히는 문제

> 태그: #트러블슈팅 #인증 #웹

## 무엇을
로그인은 성공하는데, 이후 결제 화면의 요청이 전부 인증 실패(401)로 튕겼다.

## 어떻게 풀었나
명령줄 도구(curl)로는 로그인 → 세션 갱신이 멀쩡히 되는데, 브라우저에서만 실패했다.
"요청을 누가 보내느냐"만 다르니, 범위를 브라우저 쪽으로 좁혔고 - 브라우저가 로그인 쿠키를
저장/전송하지 않는 게 원인이었다. 이유는 둘:
① 쿠키가 "보안 연결(https) 전용"으로 설정돼 있어, 로컬(http)에선 브라우저가 쿠키를 버림
② 쿠키의 교차요청 정책이 너무 엄격해 화면 흐름에서 안 실림
운영 보안은 그대로 두고, 로컬 개발에서만 이 두 속성을 완화하도록 설정으로 분리했다.

## 💡 배운 것
- "명령줄에선 되는데 브라우저에서 안 된다"면 십중팔구 쿠키(보안/교차요청) 속성 문제다.
  서버 로직을 의심하기 전에 요청 주체(curl vs 브라우저)를 갈라 비교하면 원인이 빨리 좁혀진다.
- 보안 설정은 "끄기"가 아니라 "환경별로 분리"가 정답 - 운영은 강하게, 로컬만 완화.

## 코드로 보기 (개념 재현 테스트)
쿠키 보안 속성을 "환경별로" 분리한다는 게 핵심이라, 그 판정만 자기완결 테스트로 재현한다.

```java
// 목적: 세션 쿠키의 보안 속성(Secure, SameSite)이 "환경별로" 다르게 세팅되는지 검증.
// 운영은 강하게(https 전용 + 교차요청 차단), 로컬(http)만 완화하는 게 핵심.
class SessionCookiePolicyTest {

    // 환경 설정을 흉내낸 빌더 (실제로는 프로파일/환경변수로 주입)
    private Cookie build(boolean secure, String sameSite) {
        Cookie c = new Cookie("refresh_token", "dummy-value");
        c.setHttpOnly(true);          // JS 접근 불가(XSS 방어) - 환경 무관 항상 ON
        c.setSecure(secure);          // https 전용 여부 - 환경별로 갈림
        c.setAttribute("SameSite", sameSite);  // 교차요청 전송 범위
        return c;
    }

    @Test
    void 운영환경은_https전용_교차차단() {
        Cookie c = build(true, "Strict");         // 운영: 강하게
        assertThat(c.getSecure()).isTrue();       // https 에서만 전송
        assertThat(c.getAttribute("SameSite")).isEqualTo("Strict");
    }

    @Test
    void 로컬환경은_완화해서_브라우저가_버리지_않는다() {
        Cookie c = build(false, "Lax");           // 로컬(http): 완화
        // secure=false 여야 http localhost 에서 브라우저가 쿠키를 저장/전송함
        assertThat(c.getSecure()).isFalse();
        // Lax 여야 로그인 후 화면 전환(리다이렉트) 흐름에서 쿠키가 실림
        assertThat(c.getAttribute("SameSite")).isEqualTo("Lax");
    }
}
```

## 소감
서버 코드는 멀쩡한데 왜 안 되지 하고 한참 헤맸다. curl↔브라우저를 갈라 비교하니 금방 잡혔고,
웹 기본(쿠키 속성)의 중요성을 다시 느꼈다.
