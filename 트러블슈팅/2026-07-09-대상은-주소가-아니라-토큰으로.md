# 2026-07-09 여러 대상을 넘나드는 화면에서 "지금 대상"은 주소가 아니라 토큰으로

> 태그: #트러블슈팅 #인증 #멀티테넌시 #설계

## 무엇을 / 왜
한 오너가 여러 조직(각각 독립된 데이터)을 하나의 관리 화면에서 넘나드는 구조에, "구독 결제카드
등록"을 붙이려 했다. 그런데 그대로 배선하면 등록한 카드가 지금 관리 중인 조직이 아니라 **엉뚱한
조직에 저장될 수 있는** 구조였다.

원인은 대상을 판단하는 기준이 어긋나 있던 것이다. 결제카드 등록 기능은 원래 "접속한 주소(도메인)"를
보고 대상 조직을 정하도록 만들어져 있었다. 개별 조직이 각자 자기 도메인으로 열리는 손님용 화면에선
그게 자연스럽다. 하지만 **오너의 관리 화면은 공용 주소 하나에서 여러 조직을 넘나든다.** 주소는
하나뿐이라, 주소만 봐서는 "지금 어느 조직을 고른 상태인지" 알 수가 없다. 판단 기준 자체가 이 화면엔
맞지 않았던 것이다.

## 어떻게 풀었나
대상 판단 기준을 **주소에서 토큰으로** 옮겼다.

이 관리 화면은 조직을 고르는 순간 "그 조직에만 유효한 토큰"을 갖게 된다(신원과 권한 범위를 나눈
2단계 토큰 구조). 그래서 카드 등록을 시작할 때 이 토큰에서 **대상 조직 식별자**를 꺼내, 위·변조를
막는 서명을 붙여 외부 결제창으로 실어 보냈다.

까다로운 점은 결제창이 우리 로그인 세션 **밖**에서 동작하고, 인증을 마친 결과를 우리 서버로
되돌려준다는 것이다. 이때 서버엔 로그인 세션도 없고, 돌아온 요청의 주소도 믿을 수 없다. 그래서
서버는 주소를 보지 않고, **돌아온 값에 실린 서명을 재검증해 거기서 대상 조직을 복원**한다. 서명이
맞아야만 그 조직으로 인정하므로, 중간에서 대상을 바꿔치기할 수 없다.

기존의 "주소 기반" 경로도 그대로 남겨, 서명값에 조직이 실려 있지 않으면 예전처럼 주소로 판단한다.
새 경로를 얹으면서 기존 흐름은 건드리지 않았다.

## 💡 배운 것
- **"기능이 대상을 어떻게 아는가"는 화면 성격마다 다르다.** 대상이 주소로 1:1 갈리는 화면(손님용)과,
  한 주소에서 여러 대상을 넘나드는 화면(관리용)은 판단 기준이 근본적으로 다르다. 기능을 옮겨 붙일 때
  로직만 볼 게 아니라 **그 기능이 무엇을 기준으로 대상을 정하는지**를 같이 옮겨야 한다.
- **세션 밖에서 돌아오는 외부 콜백은 요청의 출처를 믿을 수 없다.** 대상·신원 같은 핵심 정보는 나갈 때
  서명해서 실어 보내고, 돌아올 때 서명으로 복원해야 한다. 서버에 따로 저장하지 않고 **왕복만으로**
  안전하게 이어진다.
- **잘못된 대상에 민감정보가 저장되는 사고는 버그로 터지기 전에 배선 단계에서 막는 게 싸다.** 기준이
  어긋난 걸 코드 붙이기 전에 알아채면, 나중에 "왜 남의 조직에 내 카드가 있지"를 추적하는 것보다
  훨씬 값싸다.

## 코드로 보기 (개념 재현 테스트)
주소 기반 판단이 "공용 주소에서 여러 대상을 넘나드는" 화면에선 무너지고, 서명된 토큰 기반 판단은
어느 대상을 골랐든 정확히 복원되는 것을 재현한다. 서명은 개념 재현을 위해 단순화했다.

```java
// 목적: 세션 밖 외부 콜백에서 "대상"을 무엇으로 식별하느냐에 따라
//       공용 주소 화면에서 결과가 어떻게 갈리는지 재현.
class CallbackTargetResolutionTest {

    static final String SECRET = "server-only-secret";

    // 대상 식별자에 서버 비밀로 서명해 붙인다: "target.signature"
    static String sign(String target) {
        return target + "." + hmac(target, SECRET);
    }

    // 돌아온 값의 서명을 재검증해 대상을 복원한다(위조면 거부).
    static String resolveByToken(String signed) {
        int dot = signed.lastIndexOf('.');
        String target = signed.substring(0, dot);
        String sig = signed.substring(dot + 1);
        if (!hmac(target, SECRET).equals(sig)) throw new SecurityException("서명 불일치");
        return target;
    }

    // 옛 방식: 콜백이 도착한 "주소"로 대상을 정한다.
    static String resolveByHost(String host) {
        return host; // 손님용 화면에선 host == 대상. 관리 화면(공용 주소)에선 무의미.
    }

    @Test
    void 공용_주소_관리화면에서는_주소로는_고른_대상을_알_수_없다() {
        // 오너가 orgB 를 골랐지만, 관리 화면은 모두 같은 공용 주소로 열린다.
        String consoleHost = "console.example.com";
        // 주소로 판단하면 실제 고른 orgB 가 아니라 엉뚱한 값이 나온다.
        assertThat(resolveByHost(consoleHost)).isNotEqualTo("orgB");
    }

    @Test
    void 토큰으로는_어느_대상을_골랐든_콜백에서_정확히_복원된다() {
        for (String picked : new String[] {"orgA", "orgB", "orgC"}) {
            String carried = sign(picked);              // 나갈 때 서명해 실어보냄
            assertThat(resolveByToken(carried)).isEqualTo(picked); // 돌아와 복원
        }
    }

    @Test
    void 대상을_중간에_바꿔치기하면_서명검증에서_막힌다() {
        String tampered = "orgVICTIM." + hmac("orgB", SECRET); // 대상만 바꾸고 서명은 유지
        assertThatThrownBy(() -> resolveByToken(tampered))
            .isInstanceOf(SecurityException.class);
    }
}
```

## 소감
"기존 기능을 그대로 가져다 쓰면 되겠지" 하고 붙였으면, 남의 조직에 카드가 저장되는 사고가 조용히
깔릴 뻔했다. 붙이기 전에 "이 기능은 대체 무엇을 기준으로 대상을 정하지?"를 한 번 더 물은 게 컸다.
로직을 옮길 땐 로직만 옮기는 게 아니라, 그 로직이 서 있는 전제(여기선 "주소가 곧 대상")까지 같이
따라오는지 확인해야 한다는 걸 다시 새겼다.
