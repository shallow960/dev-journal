# 2026-07-03 한 곳만 막으면 다른 문으로 들어온다

> 태그: #트러블슈팅 #보안 #권한

## 무엇을 / 왜
팀 권한에 계층 규칙을 넣었다 - **자기보다 낮은 등급만 임명 가능**(오너는 관리자까지, 관리자는
직원까지). 이 규칙을 팀원 "초대" 기능에 걸어두고 끝난 줄 알았다. 그런데 다시 보니, 역할을 **직접
바꾸는 다른 경로**엔 같은 규칙이 안 걸려 있었다. 초대로는 못 하는 승격을, 직접 변경으로는 할 수
있는 상태였다. 게다가 인원 상한(좌석)도 초대 경로에서만 세고 있었다.

## 어떻게 풀었나
같은 정책을 **결과가 같은 모든 진입점**에 걸었다.
- 계층 검사와 인원 상한 검사를 **공통 함수로 빼서**, 초대·직접변경 두 경로가 같은 게이트를 쓰게 했다.
- 직접 변경은 "이미 그 역할인 사람을 다시 지정"하는 경우가 있어서, 좌석을 셀 때 **대상 본인을
  제외**하고 셌다(재지정은 새 좌석 소비가 아니므로).

이제 어느 문으로 들어와도 같은 규칙이 적용된다.

## 💡 배운 것
- **정책은 "같은 결과를 낼 수 있는 모든 경로"에 걸어야 한다.** 한 문만 잠그면 사람들은 다른 문을
  찾는다. 초대·직접변경처럼 최종 상태가 같아지는 경로들은 반드시 같은 검사를 공유해야 한다.
- **검사 로직을 흩어놓지 말고 공통화한다.** 한 경로만 고치고 다른 경로를 잊는 실수를 구조로 막는다.
- **멱등 케이스를 빼먹지 말 것.** 이미 그 역할인 사람을 다시 지정하는 건 상한에 안 부딪혀야 한다 -
  카운트에서 본인을 제외하는 한 줄로 해결됐다.

## 코드로 보기 (개념 재현 테스트)
공통 게이트를 두 경로에서 호출하고, 좌석 카운트가 본인을 제외하는지 재현한다.

```java
// 목적: 계층·좌석 규칙을 초대/직접변경 두 경로가 공유하고, 재지정은 상한에 안 걸리는지 검증.
class RoleGuardTest {

    private final int adminSeatCap = 1;
    // 현재 admin 인 사용자들
    private final Map<String, String> roleOf = new HashMap<>(Map.of("u_existing", "admin"));

    // 공통 게이트: 계층 + 좌석(대상 본인 제외 카운트)
    private void guard(String actorRole, String targetUser, String targetRole) {
        if (!canAssign(actorRole, targetRole)) {
            throw new SecurityException("hierarchy");         // 계층 위반
        }
        long used = roleOf.entrySet().stream()
            .filter(e -> e.getValue().equals(targetRole) && !e.getKey().equals(targetUser)) // 본인 제외
            .count();
        if (used >= adminSeatCap && targetRole.equals("admin")) {
            throw new SecurityException("seat-limit");        // 상한 초과
        }
    }

    private boolean canAssign(String actor, String target) {
        return actor.equals("owner") || (actor.equals("admin") && target.equals("staff"));
    }

    // 두 경로가 같은 게이트를 통과한다
    void invite(String actorRole, String user, String role)  { guard(actorRole, user, role); /* ...초대 발송 */ }
    void setRole(String actorRole, String user, String role) { guard(actorRole, user, role); /* ...역할 저장 */ }

    @Test
    void 두_경로_모두_계층을_지킨다() {
        assertThatThrownBy(() -> invite("admin", "x", "admin")).isInstanceOf(SecurityException.class);
        assertThatThrownBy(() -> setRole("admin", "x", "admin")).isInstanceOf(SecurityException.class); // 우회 불가
    }

    @Test
    void 이미_그역할이면_상한에_안걸린다() {
        // admin 상한이 1이고 u_existing 이 이미 admin 이어도, 그를 다시 admin 으로 지정하는 건 OK
        assertThatCode(() -> setRole("owner", "u_existing", "admin")).doesNotThrowAnyException();
    }
}
```

## 소감
"초대에 규칙 걸었으니 됐지" 하고 넘어갔던 게 함정이었다. 같은 결과에 이르는 경로가 둘 이상이면,
정책은 그 둘 다에 있어야 한다는 당연한 사실을 놓쳤다. 이후로는 권한 규칙을 넣을 때 "이 상태에
이르는 다른 길은 없나"부터 세어본다.
