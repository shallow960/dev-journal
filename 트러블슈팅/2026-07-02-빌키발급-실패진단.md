# 2026-07-02 정기결제 자동결제 키 발급 실패, 원인은 내 코드가 아니었다

> 태그: #트러블슈팅 #결제 #디버깅

## 무엇을
구독 자동결제를 위해 실제 카드로 "자동결제 키(빌링키)" 발급을 라이브 테스트했는데, 마지막 발급 단계에서 실패가 떴다.

## 어떻게 풀었나
실패 화면만 보면 "내 연동이 틀렸나" 싶지만, 결제사가 응답에 남긴 실패 코드와 메시지를 로그에서 먼저 확인했다.

코드를 역추적하니 실패 사유는 "입력한 카드가 테스트 환경에 등록되지 않은 카드"였다. 즉 카드 인증 -> 응답 위·변조 검증(서명) -> 발급 요청까지 내 쪽 흐름은 전부 통과했고, 결제사가 그 카드 자체를 못 받아준 것이었다.

설계상 카드번호는 내 서버를 거치지 않고 결제사 인증창에서만 다루기 때문에, "내가 만질 수 있는 구간"은 이미 정상임을 확인할 수 있었다. 유효한 실제 카드로 재시도하니 발급·저장까지 성공했다.

## 💡 배운 것
- 결제 실패는 "내 코드 / 외부 시스템 / 입력 데이터" 중 어디 책임인지부터 갈라야 한다. 외부가 돌려준 코드·메시지가 그 경계를 가르는 1차 증거다.
- 응답 코드 하나로 "전 구간 정상 + 마지막 입력 데이터만 문제"를 증명하면, 멀쩡한 코드를 뜯어고치는 헛수고를 막을 수 있다.

## 코드로 보기 (개념 재현 테스트)
"외부 응답코드로 실패 책임을 가른다"는 판정 로직을 자기완결 테스트로 재현한다.

```java
// 목적: 결제사 응답코드를 받아 "실패 책임이 어디인지" 분류하는 로직 검증.
// 실패 원인을 우리 코드/외부 시스템/입력 데이터로 갈라, 헛디버깅을 막는 게 요점.
class BillingFailureClassifierTest {

    enum Blame { OUR_CODE, INPUT_DATA, EXTERNAL }  // 실패 책임 주체

    // 결제사가 준 결과코드로 책임을 판정한다 (코드값은 예시)
    private Blame classify(String resultCode) {
        return switch (resultCode) {
            // 카드 자체 문제(미등록 카드 등) = 입력 데이터 탓, 우리 코드는 정상
            case "F112" -> Blame.INPUT_DATA;
            // 서명/인증 검증 실패 = 우리가 만든 요청이 잘못됨
            case "F400", "F401" -> Blame.OUR_CODE;
            // 그 외 = 외부 시스템 문제로 분류(재시도/문의 대상)
            default -> Blame.EXTERNAL;
        };
    }

    @Test
    void 미등록카드_실패는_우리코드가_아니라_입력데이터_탓() {
        // 라이브 테스트에서 실제로 만난 코드: "카드 BIN 없음"
        assertThat(classify("F112")).isEqualTo(Blame.INPUT_DATA);
    }

    @Test
    void 서명오류는_우리요청_문제로_잡는다() {
        assertThat(classify("F401")).isEqualTo(Blame.OUR_CODE);
    }
}
```

## 소감
실패 메시지를 보자마자 코드부터 의심했으면 한참 헤맸을 것이다. 외부가 준 코드를 먼저 읽는 습관이 시간을 크게 아꼈다. 앞으로도 외부 연동 실패는 "상대가 뭐라고 답했는지"부터 확인하고 움직일 것.
