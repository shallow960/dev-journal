# 2026-07-03 "누구인지"를 클라이언트에게 묻지 않기

> 태그: #트러블슈팅 #보안 #접근제어

## 무엇을 / 왜
운영자용 관리 화면을 손보다, 서버가 요청에 딸려 들어온 값을 그대로 믿고 있다는 걸 발견했다.
"지금 어느 조직인지 / 누구인지 / 무슨 권한인지"를 요청 헤더로 받아 그 값대로 처리하고 있었다.
즉 유효한 로그인 하나만 있으면, 그 헤더를 바꿔 **남의 조직 데이터에 더 높은 권한으로** 접근할 수 있는
상태였다. 게이트웨이가 붙여주는 값이라 "믿어도 된다"고 넘겨짚은 게 화근이었다.

## 어떻게 풀었나
세 가지를 전부 "요청이 주장하는 값"이 아니라 **서버가 검증한 것**으로만 판단하도록 바꿨다.
- 누구인지 = 서명 검증된 토큰의 주체(subject)에서.
- 어느 조직인지 = 토큰에 담긴 조직 식별자에서(헤더 아님).
- 무슨 권한인지 = 그 (사용자, 조직) 쌍으로 **저장된 소속 관계를 조회**해서.

소속 관계가 없으면 접근을 거부한다. 헤더는 더 이상 신뢰 근거가 아니게 됐다.

## 💡 배운 것
- **클라이언트가 보낸 신원·권한 값은 "사실"이 아니라 "주장"이다.** 게이트웨이가 붙여준 헤더도,
  그 뒤 경로로 직접 요청이 들어오면 얼마든지 위조될 수 있으니 그대로 신뢰하면 안 된다.
- **인가는 "요청이 뭐라고 말하는가"가 아니라 "서버가 검증한 신원 + 저장된 관계"로 판단한다.**
- **취약점은 "정상 동작 확인"이 아니라 "공격 시나리오 재현"으로 검증한다** - 남의 조직을 주장하는
  요청이 실제로 막히는지를 테스트로 못박았다.

## 코드로 보기 (개념 재현 테스트)
"요청이 주장하는 권한"이 아니라 "검증된 신원 + 저장된 소속"으로 인가하는 지점을 재현한다.

```java
// 목적: 조직/권한을 헤더로 신뢰하지 않고, 검증된 신원으로 저장된 소속을 조회해 인가하는지 검증.
class AccessControlTest {

    // 저장된 소속 관계(누가 어느 조직에서 어떤 역할인지) - 실제로는 DB 조회
    record Membership(String userId, String orgId, String role) {}

    private final List<Membership> memberships = List.of(
        new Membership("u1", "orgA", "admin")   // u1 은 orgA 의 admin 일 뿐
    );

    // 인가: 검증된 (userId, orgId)로 실제 소속을 찾는다. 없으면 접근 거부.
    private String authorize(String verifiedUserId, String verifiedOrgId) {
        return memberships.stream()
            .filter(m -> m.userId().equals(verifiedUserId) && m.orgId().equals(verifiedOrgId))
            .map(Membership::role)
            .findFirst()
            .orElseThrow(() -> new SecurityException("forbidden"));
    }

    @Test
    void 소속된_조직은_역할이_해석된다() {
        assertThat(authorize("u1", "orgA")).isEqualTo("admin");
    }

    @Test
    void 소속되지_않은_조직_접근은_거부된다() {
        // u1 이 "나 orgB 다" 라고 헤더로 주장해도, 저장된 소속에 없으면 차단(사칭 방지)
        assertThatThrownBy(() -> authorize("u1", "orgB"))
            .isInstanceOf(SecurityException.class);
    }
}
```

## 소감
"게이트웨이가 넣어준 값이니 안전하겠지"가 정확히 함정이었다. 신뢰 경계를 한 칸 안으로 당겨서,
"이 값이 어디서 왔고 누가 보증하는가"를 매번 되물으니 구멍이 보였다. 앞으로 인가 코드를 볼 땐
"이 판단의 근거가 요청이 준 값인가, 서버가 검증한 값인가"부터 확인할 것.
