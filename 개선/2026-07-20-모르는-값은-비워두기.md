# 2026-07-20 임시로 채워둔 값이 나중에 거짓말이 된다

> 태그: #개선 #설계 #정직한상태

## 무엇을 / 왜
관리 화면에서 도메인 소유 확인을 마치면 "보안 인증서 정상, 만료 90일 뒤"라고 떴다. 그런데 그 시점엔
인증서가 아직 만들어지지도 않은 상태였다. 실제 발급은 그 주소로 첫 접속이 들어올 때 일어나기 때문이다.

왜 이렇게 됐는지 찾아보니, 예전에 인증서 발급 기능이 아직 없던 시절에 화면을 먼저 만들면서 "일단
그럴듯한 값으로 채워두는" 임시 구현을 해둔 게 남아 있었다. 소유 확인이 끝나면 무조건 "정상"으로 적고,
만료일은 오늘부터 90일 뒤로 계산해 넣는 식이었다.

문제는 그 뒤에 진짜 발급 기능이 생겼다는 것이다. 이제 인증서는 다른 곳에서 실제로 발급되는데, 화면에
보이는 값은 여전히 옛날에 지어내던 값이었다. 게다가 매일 도는 자동 작업이 그 가짜 만료일을 계속
갱신까지 하고 있었다.

## 어떻게 풀었나
값을 지어내던 부분과, 매일 도는 가짜 갱신 작업을 전부 지웠다. 그리고 상태가 **실제로 일어난 일로만**
바뀌도록 다시 짰다.

- 소유 확인이 끝나면 "발급 대기"로 둔다. 자격이 생긴 것이지 인증서가 생긴 게 아니니까.
- 실제로 발급이 시작되는 순간에만 "정상"으로 바꾼다. 마침 발급 직전에 우리 쪽에 확인을 요청하는
  단계가 있어서, 그 신호를 상태 전환의 근거로 썼다.
- 만료일은 우리가 알 수 없는 값이라 아예 비워뒀다. 인증서의 수명은 발급을 맡은 쪽이 관리하고, 갱신도
  거기서 알아서 한다.

갱신 작업을 지운 것도 같은 이유다. 실제 갱신은 다른 곳에서 자동으로 하고 있는데, 우리가 매일 숫자만
새로 적어 넣는 건 아무 일도 안 하면서 "관리되고 있다"는 착각만 만들고 있었다.

## 💡 배운 것
- 기능이 없을 때 "일단 그럴듯한 값으로 채워두는" 임시 구현은, 나중에 진짜 동작이 생기면 그 자리에
  남아 **사용자에게 거짓 정보를 보여주는 코드**가 된다. 임시로 둘 때 "진짜가 들어오면 여기를 걷어낸다"를
  같이 정해둬야 한다.
- 상태 값은 실제로 일어난 일에 근거해야 한다. "이쯤이면 됐겠지"로 채운 값은 처음엔 그럴듯하지만,
  실제 동작과 어긋나는 순간 아무도 믿을 수 없는 값이 된다.
- **모르는 값은 비워두는 게 정직하다.** 채워 넣으면 화면은 풍성해 보이지만, 그 값을 보고 판단하는
  사람에게는 잘못된 정보를 주는 것이다.
- 아무 일도 하지 않으면서 도는 자동 작업은 지우는 게 낫다. 남아 있으면 "관리되고 있다"는 착각을 만들어
  진짜 문제를 늦게 발견하게 한다.

## 코드로 보기 (개념 재현 테스트)
"지어낸 값"과 "실제 신호로만 바뀌는 값"이 어떻게 다른지 더미로 재현한다.

```java
// 목적: 상태는 실제로 일어난 일로만 바뀌어야 하고, 모르는 값은 비워둬야 한다.
class HonestStateTest {

    enum CertState { PENDING, PROVISIONING, ACTIVE }

    record Domain(CertState state, LocalDate expiresAt) {}

    // 나쁜 방식: 소유 확인만 끝나면 "정상"이라 적고 만료일까지 지어낸다
    static Domain verified_bad() {
        return new Domain(CertState.ACTIVE, LocalDate.now().plusDays(90));
    }

    // 좋은 방식: 자격만 생긴 상태로 두고, 만료일은 모르니 비운다
    static Domain verified_good() {
        return new Domain(CertState.PROVISIONING, null);
    }

    // 실제 발급이 시작됐다는 신호를 받았을 때만 "정상"으로 올린다
    static Domain onIssueRequested(Domain current) {
        return new Domain(CertState.ACTIVE, current.expiresAt());
    }

    @Test
    void 소유확인만으로는_정상이라고_말하지_않는다() {
        // 지어낸 쪽 - 인증서가 없는데 정상이라고 표시된다(거짓 정보)
        assertThat(verified_bad().state()).isEqualTo(CertState.ACTIVE);
        assertThat(verified_bad().expiresAt()).isNotNull();

        // 정직한 쪽 - 아직 발급 전이므로 대기 상태, 만료일은 비어 있다
        Domain d = verified_good();
        assertThat(d.state()).isEqualTo(CertState.PROVISIONING);
        assertThat(d.expiresAt()).isNull();

        // 실제 발급 신호가 왔을 때 비로소 정상으로 바뀐다
        assertThat(onIssueRequested(d).state()).isEqualTo(CertState.ACTIVE);
    }
}
```

## 소감
"모르면 일단 채워두자"가 편해 보였는데, 결국 사용자에게 거짓말을 하는 셈이 된다는 게 이번에 제일
크게 남았다. 화면에 빈칸이 있는 게 보기 안 좋아서 뭐라도 넣고 싶어지는데, 틀린 값을 넣는 것보다
비워두는 편이 훨씬 낫다는 걸 배웠다.

임시 구현을 지울 때 "이거 지워도 되나" 싶어 관련된 곳을 전부 찾아보고 테스트를 돌려본 뒤에 지웠는데,
지우고 나니 코드가 눈에 띄게 단순해졌다. 기능을 더하는 것만큼 걷어내는 것도 개선이라는 걸 느꼈다.
