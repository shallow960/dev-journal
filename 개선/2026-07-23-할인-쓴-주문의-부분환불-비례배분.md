# 2026-07-23 할인 쓴 주문의 부분 환불 - 정가를 돌려주면 더 주게 된다

> 태그: #개선 #결제 #환불 #돈계산

## 무엇을 / 왜
가게 주인이 손님의 반품을 처리하는 기능을 만들다가, **적립금이나 쿠폰을 쓴 주문은 품목 하나만
반품하려 하면 막혀 있는** 걸 봤다. 잘못된 금액을 돌려줄 위험 때문에 "이런 주문은 전체 환불로만
하세요"로 차단해 둔 상태였다. 하지만 포인트 - 쿠폰은 손님들이 워낙 자주 써서, 사실상 대부분의
주문에서 부분 반품이 막히는 셈이라 제대로 지원하고 싶었다.

## 어떻게 풀었나
왜 막아뒀는지 파보니 함정이 있었다. 할인이 주문 **전체에 퍼져** 있는데 품목 하나를 반품하면서 그
품목의 **정가를 그대로 돌려주면, 손님이 실제로 낸 것보다 더 많은 현금을 돌려주게** 된다.

예를 들어 상품 2개(각 10,000원, 합 20,000원)를 적립금 4,000원까지 써서 샀다면 실제 카드로 낸 돈은
16,000원이다. 여기서 한 품목을 반품하며 정가 10,000원을 돌려주고, 다른 품목도 그렇게 하면 합이
20,000원 - 실제 낸 16,000원보다 많다. 4,000원을 공짜로 얹어주는 셈이다.

그래서 **각 품목이 받은 할인 몫을 비율대로 나눠서** 처리했다.

- 품목의 **현금 환불 = 품목값 - (그 품목이 나눠 받은 적립금 - 쿠폰 몫)**. 위 예에서 한 품목은
  적립금 몫 2,000원을 빼고 **현금 8,000원**만 돌려준다.
- 쓴 **적립금은 그 품목 몫(2,000원)만큼 되돌려**준다.
- **쿠폰 몫은 되돌려주지 않는다.** 쿠폰은 한 번 쓰면 끝이고, 남은 품목엔 그 혜택이 그대로 남기
  때문이다.
- 특히 나눌 때 생기는 소수점을 **항상 내림(덜 주는 쪽)** 으로 처리했다. 그래야 여러 품목을 합쳐도
  **실제 낸 돈 - 쓴 적립금을 절대 넘지 않는다.**

적립금 복원도 손봤다. 그냥 "쓴 만큼 돌려준다"로 두면 부분 환불과 전체 환불이 섞였을 때 두 번
돌려줄 수 있다. 그래서 **"이 주문에서 이미 얼마를 복원했는지"를 세어 나가게** 바꿔서, 부분 - 전체가
섞여도 **돌려준 적립금 합이 쓴 적립금을 넘지 못하게** 했다.

상품 2개를 적립금까지 써서 산 주문에서 한 품목만 반품하면, 그 품목 몫의 현금(8,000원)과
적립금(2,000원)이 정확히 돌아오는 것을 테스트로 확인하고, 전체 검사를 통과시킨 뒤 배포했다.

## 💡 배운 것
- **할인이 주문 전체에 퍼져 있으면, 품목 환불은 "정가"가 아니라 "그 품목이 실제 부담한 몫"으로 해야
  한다.** 정가로 돌려주면 퍼져 있던 할인만큼 과다 환불이 된다.
- **돈을 나눌 때 반올림 방향은 "절대 더 주지 않는 쪽(내림)"으로 고정한다.** 방향 하나로 "합쳐도
  넘지 않음"이 보장되어, 여러 품목을 개별 환불해도 총액이 안전하다.
- **넘지 말아야 할 선(불변식)을 먼저 정하고 지킨다.** "총 환불 <= 실제 낸 돈", "돌려준 적립금 <= 쓴
  적립금" 을 정해두고, 복원은 이미 돌려준 양을 누적으로 세어 그 선을 강제했다.
- 막아둔 기능을 열 땐 **왜 막았는지(위험)를 먼저 이해**하고 그 위험을 없앤 뒤 열어야 한다. 위험을
  안 없애고 그냥 열면 막아둔 이유가 그대로 사고가 된다.

## 코드로 보기 (개념 재현 테스트)
"정가 환불은 과다, 비례 배분 + 내림은 실제값을 넘지 않음"을 개념만 재현한 골격이다. 실제 내부
이름 - 테이블은 뺐고 값은 더미다.

```java
// 목적: 할인이 주문 전체에 퍼진 경우, 품목 환불을 "정가"로 하면 과다환불이 되고
//       "그 품목이 실제 부담한 몫"으로 하면 실제 낸 돈을 넘지 않음을 재현.
//       핵심: 몫은 비율로 나누되 "내림", 합계는 실제 낸 돈/쓴 적립금을 넘지 못한다.
class ProrateRefundTest {

    // 주문: 상품 2개 각 10,000 (합 20,000), 적립금 4,000 사용 -> 실제 낸 돈 16,000
    static final long ITEM = 10_000, TOTAL = 20_000, POINTS_USED = 4_000;

    // 한 품목의 현금 환불 = 품목값 - (적립금 몫, 비율대로 내림)
    static long cashRefund(long itemGross) {
        long pointShare = POINTS_USED * itemGross / TOTAL; // 정수 나눗셈 = 내림
        return itemGross - pointShare;                     // 쿠폰 몫이 있으면 같은 방식으로 더 뺀다
    }
    // 되돌릴 적립금 = 그 품목 몫(내림). 쿠폰 몫은 1회성이라 되돌리지 않는다.
    static long pointRestore(long itemGross) {
        return POINTS_USED * itemGross / TOTAL;
    }

    @Test
    void 정가로_돌려주면_실제_낸_돈보다_많이_준다() {
        long naive = ITEM + ITEM;                              // 두 품목 정가 합 = 20,000
        assertThat(naive).isGreaterThan(TOTAL - POINTS_USED);  // 16,000 보다 크다 = 과다환불
    }

    @Test
    void 비례배분_내림이면_현금과_적립금이_실제값을_넘지_않는다() {
        long cash = cashRefund(ITEM) + cashRefund(ITEM);
        long restore = pointRestore(ITEM) + pointRestore(ITEM);
        assertThat(cash).isLessThanOrEqualTo(TOTAL - POINTS_USED); // 현금 합 <= 실제 낸 돈
        assertThat(restore).isLessThanOrEqualTo(POINTS_USED);      // 복원 합 <= 쓴 적립금

        // 한 품목만 반품하면: 현금 8,000 + 적립금 2,000 복원
        assertThat(cashRefund(ITEM)).isEqualTo(8_000);
        assertThat(pointRestore(ITEM)).isEqualTo(2_000);
    }
}
```

## 소감
"그냥 그 품목값 돌려주면 되지" 가 함정이었다. 할인이 주문 전체에 퍼져 있으면 정가 환불은 곧 과다
환불이라는 걸, 예시 숫자를 직접 짚어보고서야 손에 잡혔다. 돈 계산에선 "적당히 맞게"가 아니라
"어느 방향으로 틀려도 손해가 안 나는 쪽"으로 규칙을 잡아야 안심이 된다는 걸 배웠다. 그래서 반올림도
내림으로 고정하고, "넘지 말아야 할 선"을 못 박아둔 뒤에야 막아뒀던 기능을 열 수 있었다.
