# 2026-07-03 기존 사용자를 안 막고 권한 제한을 켜기

> 태그: #개선 #접근제어 #롤아웃

## 무엇을 / 왜
운영 화면에 "직원" 등급은 민감한 기능(결제·정산·설정 등)을 못 만지게 제한하려 했다. 문제는
이미 돌아가는 시스템에 새 제한을 얹을 때 늘 따라오는 위험 - 잘못 켜면 **기존 운영자·오너까지
막혀버린다(락아웃)**. 권한을 조이는 변경은 새 기능 추가보다 사고가 잦다.

## 어떻게 풀었나
제한을 "최소한으로, 안전한 기본값"으로 설계했다.
- **허용 목록이 아니라 차단 목록(블록리스트)으로.** 직원이 못 가는 민감 영역만 나열하고, 그 외는
  전부 허용. 나중에 새 화면이 추가돼도 자동으로 막히지 않는다(기본이 허용).
- **명시적으로 "직원"인 사람만 제한.** 역할 정보가 없는 기존 사용자나 오너·관리자는 아예 건드리지
  않고 통과시킨다.
- 결과적으로 제한은 오직 "직원 × 민감 영역"에만 걸리고, 나머지 흐름은 이전과 100% 똑같다.

## 💡 배운 것
- **새 권한 제한은 "누가 막히나"보다 "누가 안 막히나(기본값)"를 먼저 정한다.** 화이트리스트는
  빠뜨린 하나가 락아웃, 블록리스트는 빠뜨린 하나가 과허용이다. **과허용은 나중에 조이면 되지만
  락아웃은 즉시 사고**라, 롤아웃 안전성은 블록리스트가 높다.
- **역할 정보가 없을 때의 기본값을 "허용"으로.** 데이터가 아직 안 채워진 사용자를 실수로 막지 않는다.
- **제한 범위를 최소로.** "직원에게만, 민감 영역에만" 걸어서, 나머지는 검증할 필요조차 없게 했다.

## 코드로 보기 (구조 골격)
"차단 목록 + 직원만 제한 + 기본 허용"의 판정 골격을 재현한다.

```java
// 목적: 새 권한 제한을 기존 사용자 락아웃 없이 거는 판정 로직.
class StaffGateTest {

    // 직원이 접근 못 하는 영역만 나열(그 외는 전부 허용) - 화이트리스트가 아니라 블록리스트
    private static final List<String> BLOCKED_FOR_STAFF = List.of("billing", "settlement", "settings");

    // role 은 "직원"일 때만 존재한다고 가정(그 외/미등록은 null)
    private boolean allowed(String role, String area) {
        if (!"staff".equals(role)) {
            return true;                          // 오너·관리자·미등록 사용자는 무제한(락아웃 방지)
        }
        return !BLOCKED_FOR_STAFF.contains(area); // 직원은 차단 목록만 거부
    }

    @Test
    void 직원은_민감영역만_막힌다() {
        assertThat(allowed("staff", "billing")).isFalse();  // 결제 = 차단
        assertThat(allowed("staff", "orders")).isTrue();    // 주문 = 허용(목록에 없음)
    }

    @Test
    void 오너나_미등록_사용자는_제한없다() {
        assertThat(allowed("owner", "billing")).isTrue();
        assertThat(allowed(null, "billing")).isTrue();      // 역할 정보 없어도 안 막힘
    }
}
```

## 소감
"권한을 조인다"고 하면 막는 것만 생각하기 쉬운데, 실전에선 "누구를 안 막을지"가 더 중요했다.
기본값을 허용으로, 범위를 최소로 두니 배포가 겁나지 않았다. 조이는 변경일수록 되돌릴 수 있는
방향(과허용)으로 실수하게 설계하는 게 안전하다는 걸 배웠다.
