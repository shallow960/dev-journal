# 2026-07-14 부가 기능을 독립 청구로 쪼개 환불을 깔끔하게

> 태그: #개선 #결제 #설계

## 무엇을 / 왜
구독에 부가 기능을 추가로 결제해 얹는 기능을 만들었다. 어려운 건 결제가 아니라 취소·환불이었다.
부가 기능을 구독 요금과 한 건으로 묶어 청구하면, 나중에 그중 하나만 해지할 때 "그 몫이 얼마였는지"를
한 건에서 떼어내 되돌리기가 어렵다. 금액과 사용 기간을 사후에 짜맞춰야 해서 오차와 분쟁 소지가 생긴다.

## 어떻게 풀었나
부가 기능을 구독 요금과 별개의 독립 청구 건으로 분리했다. 각자 고유한 결제 식별자를 갖게 하니,
- 특정 기능을 해지하면 정확히 그 건만 부분취소하면 되고, 구독 본체나 다른 기능의 청구 주기는 손대지 않는다.
- 주기 중간에 켜면 남은 기간만큼 일할로 즉시 청구하고, 다음 주기부터 월 요금에 합산한다.
- 해지 환불은 요금제 중도해지와 같은 일할 계산 규칙을 그대로 재사용했다.

핵심은 "환불을 어떻게 계산하나"가 아니라 "환불 대상을 어떻게 분리해 두나"였다. 청구 단위를 미리
쪼개 두면 환불은 그 건 하나를 취소하는 단순한 일이 된다.

## 💡 배운 것
- 취소·환불이 쉬우려면 청구를 미리 쪼개 둔다. 나중에 한 덩어리에서 일부를 떼어내는 것보다,
  처음부터 되돌릴 단위로 나눠 청구해 두는 게 훨씬 안전하다.
- "정확히 계산"보다 "정확히 분리"가 먼저다. 금액 계산 로직을 정교하게 만들기 전에, 되돌릴 대상의
  경계를 데이터 모델에서 그어두면 계산 자체가 단순해진다.
- 요구사항("부분 환불을 정확히")의 숨은 함정(합산 청구 시 분리 불가)은 로직이 아니라 설계로 푼다.

## 코드로 보기 (개념 재현 테스트)
합산 청구 대신 독립 청구로 두면 "특정 항목만 취소"가 얼마나 단순해지는지 더미로 보인다(실제 금액·거래 아님).

```java
// 목적: 부가기능을 독립 청구 건으로 두면, 취소가 "그 건 하나 무효화"로 끝남을 보인다.
class AddonBillingTest {

    // 청구 건 하나 = 고유 id + 금액 + 취소 여부
    record Charge(String id, long amount, boolean cancelled) {
        Charge cancel() { return new Charge(id, amount, true); }
    }

    @Test
    void 부가기능_해지는_그_건만_취소하고_나머지는_그대로() {
        // 구독 본체와 부가기능을 각각 독립 청구 건으로 발행
        Charge plan  = new Charge("plan",  10_000, false);
        Charge addon = new Charge("addon",  3_000, false);

        // 부가기능만 해지 -> 그 건 하나만 취소
        Charge cancelledAddon = addon.cancel();

        assertThat(cancelledAddon.cancelled()).isTrue();
        assertThat(plan.cancelled()).isFalse();          // 구독 본체는 무영향
        // 되돌릴 금액을 한 건에서 짜맞출 필요가 없다 -> 그 건 금액이 곧 환불 대상
        assertThat(cancelledAddon.amount()).isEqualTo(3_000);
    }
}
```

## 소감
"환불을 정확히"라는 요구를 받으면 계산식부터 파고들기 쉬운데, 이번엔 한 발 물러서서 "그럼 되돌릴
대상을 어떻게 나눠 둘까"를 먼저 정한 게 주효했다. 청구를 미리 쪼개 두니 환불은 특별한 계산 없이
그 건 하나를 취소하는 일이 됐다. 어려운 문제를 계산이 아니라 구조로 옮겨 푼 느낌이었다.
