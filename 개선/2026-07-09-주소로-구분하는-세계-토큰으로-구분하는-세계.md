# 2026-07-09 주소로 구분하는 세계, 토큰으로 구분하는 세계

> 태그: #개선 #보안 #멀티테넌시 #설계

## 무엇을 / 왜
한 시스템 안에 성격이 정반대인 두 종류의 화면이 공존했다.

- **손님용 몰**: 조직마다 자기 도메인이 있다. "어느 조직이냐"가 **접속 주소로 1:1** 갈린다.
- **오너의 관리 콘솔**: 한 오너가 여러 조직을 **공용 주소 하나에서** 넘나든다. 주소는 하나뿐이라,
  "지금 어느 조직을 다루는 중이냐"를 주소로는 알 수 없다.

두 세계가 섞이면서 두 가지 문제가 드러났다. 하나는 "지금 대상이 무엇이냐"의 판단 기준이 화면마다
달라 혼선이 생긴 것. 다른 하나는 더 위험했다 - **손님 몰 주소로 들어와도 관리·운영 화면이 그대로
열리고 있었다.** 요청을 가장 먼저 거르는 길목 검사가 몇몇 주소에서만 돌도록 돼 있어서, 그 검사망을
비껴가는 주소들(관리 화면 등)은 어느 몰 도메인으로 접근하든 무방비로 노출됐다.

## 어떻게 풀었나
두 세계의 규칙을 명확히 갈랐다.

- **대상 식별 기준을 세계별로 통일했다.** 손님용 화면은 예전처럼 주소로 조직을 가린다. 반대로 여러
  조직을 넘나드는 관리 콘솔은, 조직을 고르는 순간 발급되는 **"그 조직 전용 토큰"** 을 유일한 대상
  기준으로 삼는다. 주소로 판단하는 코드와 토큰으로 판단하는 코드가 섞이지 않게 분리했다.
- **경계를 허용 목록으로 막았다.** 손님 몰 주소로는 몰과 관련된 경로만 통과시키고, 관리·운영 화면
  경로는 그 주소에선 "존재하지 않는 것"으로 취급해 막았다. 그리고 길목 검사가 몇몇 주소만이 아니라
  **모든 경로**에서 돌도록 넓혔다. 몰 주소에선 몰만 보이고, 관리 화면은 제자리(플랫폼 주소)에서만
  열린다.

## 💡 배운 것
- **한 시스템에 "주소로 구분하는 세계"와 "토큰으로 구분하는 세계"가 같이 살면, 각 세계의 식별 기준을
  명확히 분리해야 한다.** 한쪽 세계의 기준(주소=대상)을 다른 세계에 그대로 적용하면 조용히 어긋난다.
  기능을 옮겨 붙일 때 "이 기능은 무엇을 기준으로 대상을 정하지?"를 매번 다시 물어야 한다.
- **길목 검사는 일부 경로만 걸면 다른 문으로 들어온다.** 검사망을 몇몇 주소에만 두면, 목록에 없는
  경로가 곧 뒷문이 된다. 격리는 "막을 것을 나열"이 아니라 "허용할 것만 나열하고 나머지는 전부 차단"
  으로 가야 빠짐이 없다.
- **경계는 눈에 안 보여서 더 위험하다.** 화면이 멀쩡히 떠 있으니 아무도 "이게 여기서 열리면 안 되는데"를
  눈치채지 못한다. 노출은 터지기 전엔 티가 안 나므로, 구조를 그릴 때 "이 주소에서 이게 열려도 되나"를
  의식적으로 점검해야 한다.

## 코드로 보기 (구조 골격)
"몰 주소에선 허용 경로만 통과, 나머지는 없는 것으로 취급"하는 길목 검사와, 관리 콘솔이 주소 대신
토큰으로 대상을 정하는 분리를 재현한다.

```ts
// 목적: 두 세계의 규칙을 분리 - 손님 몰 주소는 허용 목록으로 격리,
//       관리 콘솔은 주소가 아니라 "고른 조직 토큰"으로 대상을 정한다.

const MALL_ALLOWED = ["/shop", "/cart", "/api"]; // 몰 주소에서 허용할 경로(허용 목록)

// 요청 길목 검사: 모든 경로에서 돈다.
function guard(host: string, path: string) {
  if (isMallHost(host)) {
    // 몰 주소: 허용 목록에 없는 경로(관리·운영 화면 등)는 "없는 것"으로 취급.
    const allowed = MALL_ALLOWED.some((p) => path === p || path.startsWith(p + "/"));
    return allowed ? "pass" : "not-found"; // 뒷문 없음
  }
  return "pass"; // 플랫폼 주소: 관리 화면 정상 접근
}

// 관리 콘솔의 대상 판단은 주소가 아니라 "고른 조직 토큰"에서 온다.
function consoleTargetOrg(storeToken: { orgId: string }) {
  return storeToken.orgId; // 공용 주소여도 토큰이 대상을 특정한다
}

// 검증
test("몰 주소에선 관리 화면 경로가 막힌다", () => {
  expect(guard("shopA.example.com", "/admin")).toBe("not-found");
  expect(guard("shopA.example.com", "/shop/123")).toBe("pass");
});
test("플랫폼 주소에선 관리 화면이 열린다", () => {
  expect(guard("console.example.com", "/admin")).toBe("pass");
});
```

## 소감
"콘솔에서 카드가 엉뚱한 조직에 저장될 뻔한 문제"를 파고들다 보니, 근본은 카드가 아니라 **두 세계가
서로 다른 규칙으로 대상을 가린다는 사실**이었다. 주소로 사는 화면과 토큰으로 사는 화면을 같은 잣대로
다루려던 게 화근이었다. 두 세계를 인정하고 각자의 규칙을 명확히 갈라주니, 대상 혼선도 노출도 함께
정리됐다. 시스템이 커질수록 "여기선 무엇으로 대상을 정하는가"를 세계 단위로 의식하게 됐다.
