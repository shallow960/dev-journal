# 2026-07-09 만들지 않고 이어붙이기 - 커스텀 홈과 작동 커머스의 결합

> 태그: #개선 #설계 #재사용

## 무엇을 / 왜
쇼핑몰에는 성격이 다른 두 화면이 따로 있었다. 하나는 오너가 블록을 골라 자유롭게 꾸미는 "홈",
다른 하나는 실제로 물건이 담기고 결제되는 "커머스 화면(상품·장바구니·결제)". 이 둘을 하나로 이어
붙여야 했다.

흔한 접근은 커머스까지 블록으로 다시 만드는 것이다. 그래야 홈부터 결제까지 한 방식으로 통일되니까.
하지만 이미 잘 도는 커머스를 재구현하는 건 낭비이자 위험이었다. 결제·재고·주문 같은 검증된 흐름을
새로 짜면, 얻는 건 "일관성" 하나인데 잃을 수 있는 건 그 전부다.

## 어떻게 풀었나
커머스는 그대로 두고, 손님이 보는 몰의 **첫 화면(홈)만** 블록 조립 방식으로 바꿨다.

- **새 주소를 따로 파지 않았다.** 기존 몰 홈이 뜨던 그 자리의 내용만 블록 렌더로 교체했다. 그래서
  몰이 공유하는 공통 틀(상단바·장바구니·하단바·테마)과 상세·장바구니·결제 흐름을 손대지 않고 그대로
  물려받았다.
- **블록의 링크는 기존 커머스 화면으로 이어진다.** 홈에서 상품·카테고리를 누르면 새로 만든 게 아니라
  원래 있던 상세·목록 화면으로 자연스럽게 넘어간다.
- **장식용 껍데기였던 블록에 실제 데이터를 채웠다.** 원래 홈 블록들은 회색 네모로 자리만 잡은
  껍데기였는데, 이미 커머스가 쓰던 상품·카테고리 조회를 그대로 재사용해 진짜 데이터로 채웠다. 데이터가
  없으면 어색한 빈 화면 대신 자연스러운 기본 화면으로 대체했다.

## 💡 배운 것
- **새로 만드는 게 늘 정답은 아니다.** "홈부터 결제까지 한 방식으로 통일"은 그럴듯하지만, 그 대가가
  검증된 커머스를 다시 짜는 거라면 얻는 것보다 잃는 게 크다. 통일성보다 "이미 도는 걸 안 건드리는
  것"의 가치가 더 클 때가 있다.
- **이어 붙이려면 경계를 정확히 그어야 한다.** "홈은 자유 편집 / 파는 화면은 검증된 그대로"라는 경계를
  분명히 하니, 교체할 지점(홈 내용)과 재사용할 지점(공통 틀·커머스 흐름)이 딱 갈렸다. 최소 지점만
  갈아끼우면 되니 리스크도 최소가 됐다.
- **재사용은 "그대로 쓰기"가 아니라 "빈 곳만 채우기"다.** 껍데기 블록에 이미 있던 데이터 조회를 꽂아
  넣은 것처럼, 없는 조각만 정확히 찾아 메우면 대부분은 이미 있는 걸로 굴러간다.

## 코드로 보기 (구조 골격)
"홈은 블록으로 새로 조립하되, 나머지 화면과 같은 껍데기(공통 틀)를 공유하고 링크는 기존 화면으로
간다"는 골격을 재현한다.

```tsx
// 목적: 커머스를 재구현하지 않고, 홈 본문만 블록 렌더로 교체하면서
//       공통 틀·기존 커머스 링크를 그대로 물려받는 구조를 재현.

// 몰 전체가 공유하는 껍데기(상단바·장바구니·하단바·테마). 홈이든 상세든 이걸 두른다.
function StoreShell({ children }) {
  return (
    <div className="store-theme">
      <Header /> {children} <Footer />
    </div>
  );
}

// 홈: 블록을 조립해 새로 그린다. 단, 링크는 "기존 커머스 화면" 경로로 나간다(새로 안 만듦).
function Home({ blocks, products }) {
  return blocks.map((b) =>
    b.type === "goods"
      ? products.map((p) => (
          <a key={p.id} href={`/goods/${p.id}`}>{p.name}</a> // ← 원래 있던 상세 화면으로
        ))
      : <Banner key={b.id} config={b.config} />
  );
}

// 상세·장바구니·결제: 손대지 않은 기존 커머스 화면 그대로.
function ProductDetail() { /* 재구현하지 않음 - 원래 코드 그대로 */ }

// 결론: 홈만 새로 조립되고, 껍데기와 파는 화면은 통째로 재사용된다.
//   <StoreShell><Home .../></StoreShell>  /  <StoreShell><ProductDetail/></StoreShell>
```

## 소감
"통일하려면 다 새로 짜야 한다"는 생각이 먼저 들었는데, 한 발 물러서 보니 통일이 목적이 아니라
"손님이 이음매를 못 느끼는 것"이 목적이었다. 그러면 파는 화면까지 갈아엎을 이유가 없었다. 새 기능을
많이 만드는 것보다, 안 만들어도 되는 걸 알아보고 이미 있는 걸 정확히 이어 붙이는 판단이 더 어렵고
더 값지다는 걸 느꼈다.
