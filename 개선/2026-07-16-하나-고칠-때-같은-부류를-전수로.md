# 2026-07-16 취약점 하나를 고칠 때, 같은 부류를 전수로 닫기

> 태그: #개선 #보안 #방법

## 무엇을 / 왜
전에 "남의 정보를 볼 수 있던 문제(권한 우회)"를 한 번 고친 적이 있다. 이번에 문득, 같은 종류의
허점이 다른 화면에도 흩어져 있을 것 같았다. 발견될 때마다 하나씩 땜질하기보다, 아예 "이런 부류"라고
정의해 놓고 관련된 곳을 전부 훑어 한 번에 닫아보기로 했다.

## 어떻게 풀었나
먼저 부류를 한 문장으로 정의했다. "서버가 사용자를 판별할 때, 사용자가 스스로 보내는 값을 그대로 믿는
곳." 이런 값은 얼마든지 위조할 수 있어서, 로그인 증표(토큰)에서 신원을 꺼내야 안전하다.
- 정의에 해당하는 표면을 코드에서 전수로 찾았다. 요청 헤더나 요청 본문에 담긴 "내가 누구다" 값을
  그대로 쓰는 곳들을 목록으로 만들었다.
- 목록의 각 지점을, 이미 다른 곳에서 검증된 방식(로그인 증표에서 신원을 꺼내는 방식)으로 똑같이 바꿨다.
  인증이 아예 없던 경로에는 인증부터 걸었다.
- "위조한 값을 보내도 무시되고, 증표의 주인으로만 처리된다"를 자동 테스트로 확인했다.
- 마지막으로 "이 부류에 해당하는 곳이 이제 0인가"를 다시 훑어 마무리했다.

특징적이었던 건, 대부분 대단한 새 방어를 만드는 일이 아니라 이미 만들어 둔 안전한 방식을 "이 경로에도
연결"하기만 하면 되는 누락형이었다는 점이다. 그래서 수정 비용이 생각보다 낮았다.

## 💡 배운 것
- 취약점은 "한 건"이 아니라 "부류"로 보는 게 낫다. 하나를 고치면 "이거랑 같은 게 또 어디 있지?"를
  끝까지 따라가야 진짜로 닫힌다.
- 보안은 멋진 방어를 새로 발명하는 일보다, 이미 있는 방어를 빠짐없이 연결하는 일일 때가 많다. 문제는
  "발명"이 아니라 "누락"인 경우가 흔하다.
- "다 닫았다"를 느낌으로 끝내지 말고, 부류의 정의로 다시 전수 조사해 남은 개수가 0인지 확인해야
  마무리가 된다.

## 코드로 보기 (개념 재현 테스트)
"사용자가 보낸 값"이 아니라 "증표의 주인"으로 처리해야, 위조 요청이 통하지 않는다는 걸 더미로 재현한다.

```java
// 목적: 신원을 요청 헤더(위조 가능)가 아니라 로그인 증표에서 꺼내야 안전하다.
class IdentityFromTokenTest {

    // 요청에 담긴 "내가 누구다" 값은 공격자가 마음대로 넣을 수 있다
    record Request(String headerUserId, String tokenSubject) {}

    // 나쁜 방식: 요청 헤더를 그대로 신뢰
    static String whoAmI_bad(Request req) { return req.headerUserId(); }

    // 좋은 방식: 검증된 증표의 주인만 신뢰 (헤더는 무시)
    static String whoAmI_good(Request req) { return req.tokenSubject(); }

    @Test
    void 헤더를_위조해도_증표의_주인으로만_처리된다() {
        // 공격자(A)가 헤더엔 피해자(B)라고 적어 보냈지만, 증표는 A 것이다
        Request forged = new Request("victim-B", "attacker-A");

        assertThat(whoAmI_bad(forged)).isEqualTo("victim-B");    // 헤더 신뢰 - 위조 성공(위험)
        assertThat(whoAmI_good(forged)).isEqualTo("attacker-A"); // 증표 신뢰 - 위조 무력화(안전)
    }
}
```

## 소감
한 곳을 고치고 나서 "비슷한 게 또 있지 않을까"를 흘려보내지 않은 게 이번의 핵심이었다. 부류로 정의해
놓으니 어디를 봐야 할지 분명해졌고, 다 훑고 나서 남은 개수가 0인 걸 확인했을 때 비로소 "닫혔다"는
확신이 들었다. 보안은 화려한 한 방보다, 같은 실수를 빠짐없이 쓸어 담는 꼼꼼함에 가깝다는 걸 느꼈다.
