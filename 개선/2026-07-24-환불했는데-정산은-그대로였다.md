# 2026-07-24 환불했는데 판매자 정산은 그대로였다

> 태그: #개선 #돈계산 #정합성 #상태머신

## 무엇을 / 왜

환불 로직을 손보다가 궁금해졌다. 손님이 물건을 받고 "구매확정"을 하면 그 판매 대금이 **판매자에게
정산될 돈으로 잡히는데**, 구매확정 뒤에도 반품이 가능하다. 그럼 반품해서 손님에게 환불할 때 판매자
정산도 같이 취소되나?

코드를 따라가 보니 환불하는 어디에도 정산을 되돌리는 부분이 없었다. 손님은 돈을 돌려받는데 판매자
정산은 그대로 남는다. 판매자에게 실제로 송금하는 기능은 아직 계약 전이라 사고는 안 났지만, 그게
켜지는 순간 **같은 판매에 손님 환불 + 판매자 지급이 이중으로** 나갈 상황이었다.

## 어떻게 풀었나

먼저 정산에 "취소" 상태를 더하고, 반품이 완료되면 그 건의 정산을 되돌리게 했다. 아직 지급 전이면
취소, 이미 지급 절차에 들어갔으면 함부로 되돌리지 않고 "보류"로 잡아 사람이 확인하게.

그런데 "지급 절차에 들어갔는지"를 정산 상태로 판단하려는데, **그 상태가 실제로는 갱신되지 않는다**는
걸 발견했다. 출금을 요청하거나 승인해도 정산 상태가 그대로였다. "확정 → 출금요청 → 지급완료"라는
상태 흐름이 정의만 돼 있고 **아무도 그 상태를 움직이지 않아 죽어 있던** 것이다.

죽은 상태 흐름은 곧바로 다른 구멍이었다. 상태가 안 바뀌니 같은 정산에 **출금을 여러 번 요청**할 수
있었다(관리자가 둘 다 승인하면 또 이중 지급). 이건 고치기 전 코드에서 테스트로 직접 재현해 뚫리는
것을 확인한 뒤, 출금 요청이 정산 상태를 바꾸도록(두 번째 요청은 거부), 승인은 지급완료로, 거부는
다시 요청 가능하게 흐름을 이어 붙였다. 그러자 처음 만든 회수 로직도 비로소 정확해졌다 - 출금 요청된
정산은 이제 "요청됨" 상태라 회수가 "보류"로 잡고, 요청 전이면 "취소"로 처리한다.

## 💡 배운 것

- **돈이 한 곳에서 빠지면 연결된 다른 곳에서도 되돌아가야 한다.** 환불(손님)만 있고 회수(판매자)가
  없으면 장부가 한쪽으로 기운다. "이 돈이 빠질 때 같이 움직여야 할 게 또 있나"를 물어야 한다.
- **아직 안 켜진 기능이라도 켜지기 전에 구멍을 막는다.** 지급이 계약 전이라 지금은 사고가 없지만,
  켜는 순간이 곧 사고다. latent 결함은 실가동 전에 닫는 게 싸다.
- **상태 흐름이 정의만 되고 아무도 안 움직이면 죽은 코드다.** 상태 머신이 있다고 안심하면 안 되고,
  각 전이를 실제로 누가 일으키는지 확인해야 한다.
- **한 곳을 고치다 드러난 인접 버그를 놓치지 않는다.** 회수 로직이 참조한 상태가 갱신 안 된다는 사실이
  중복 출금이라는 더 근본적인 버그로 이어졌다.

## 코드로 보기 (개념 재현 테스트)

"출금 요청이 상태를 바꾸지 않으면 중복 요청이 통과하고, 상태를 전이시키면 막힌다"를 개념만 재현했다.
실제 내부 이름은 뺐고 값은 더미다.

```java
// 목적: 출금 요청이 정산 상태를 "요청됨"으로 바꾸지 않으면 같은 정산에 여러 번 요청이 통과하고
//       (관리자 이중 승인 -> 이중 지급), 상태를 전이시키면 두 번째 요청이 막힌다.
class WithdrawalStateTest {

    enum SettlementState { CONFIRMED, REQUESTED, PAID }   // 확정 -> 출금요청 -> 지급완료

    static class Settlement {
        SettlementState state = SettlementState.CONFIRMED;
        int withdrawalCount = 0;
    }

    // (취약) 상태를 안 바꾼다. CONFIRMED 든 REQUESTED 든 다 허용 -> 몇 번이고 요청 가능.
    static void requestWithdrawalBuggy(Settlement s) {
        if (s.state != SettlementState.CONFIRMED && s.state != SettlementState.REQUESTED) {
            throw new IllegalStateException("not withdrawable");
        }
        s.withdrawalCount++;                 // 출금 요청 생성
        // s.state 를 바꾸지 않는다 = 다음 요청도 같은 조건을 통과한다
    }

    // (수정) CONFIRMED 에서만 허용하고, 요청하면 REQUESTED 로 전이한다.
    static void requestWithdrawalFixed(Settlement s) {
        if (s.state != SettlementState.CONFIRMED) {
            throw new IllegalStateException("not withdrawable");
        }
        s.withdrawalCount++;
        s.state = SettlementState.REQUESTED; // 다음 요청은 CONFIRMED 가 아니라서 막힌다
    }

    @Test
    void 상태를_안_바꾸면_중복_출금이_통과한다() {
        Settlement s = new Settlement();
        requestWithdrawalBuggy(s);
        requestWithdrawalBuggy(s);           // 막히지 않는다
        assertThat(s.withdrawalCount).isEqualTo(2); // 출금 2건 = 이중 지급 위험
    }

    @Test
    void 요청이_상태를_전이하면_두번째는_막힌다() {
        Settlement s = new Settlement();
        requestWithdrawalFixed(s);
        assertThatThrownBy(() -> requestWithdrawalFixed(s)) // 이미 REQUESTED
            .isInstanceOf(IllegalStateException.class);
        assertThat(s.withdrawalCount).isEqualTo(1);
    }
}
```

## 소감

"환불되나?"라는 단순한 질문에서 시작했는데, 파고들수록 문제가 한 겹씩 벗겨졌다. 환불이 정산으로
이어지지 않던 것 → 그걸 고치려 본 상태가 실은 죽어 있던 것 → 그래서 중복 출금까지 되던 것. 회수만
고치고 멈췄으면 "출금 진행분은 보류" 로직이 상태를 잘못 읽어 조용히 틀렸을 것이다. 고치다 만난
"어? 이 상태 왜 안 바뀌지"를 그냥 넘기지 않은 게 컸다. 그리고 중복 출금은 고치기 전에 먼저 재현해
두었다 - 돈이 걸린 수정일수록 "정말 뚫렸었나"를 눈으로 봐야 안심이 된다.
